# Aerocool Content SEO Checklist 2026

Актуально на `2026-04-21`.

Этот документ — единый editorial и SEO SOP для всего проекта `Aerocool`. Он адаптирован под текущую структуру сайта на `Hugo` с разделами `home`, `products`, `articles`, `news`, `faq`, `about`, `contact`, `search`, а также под двуязычную архитектуру `uk` + `ru`.

Важно: ни один чек-лист не гарантирует `топ-1` в Google. По официальным рекомендациям Google, даже если страница соблюдает Search Essentials и лучшие практики, это не означает гарантированный crawl, index или ranking. Цель этого документа — максимально повысить вероятность сильного ранжирования, CTR и коммерческой полезности.

## 2. Неподвижные Правила Для Любой Страницы

- [ ] У каждой URL должен быть один главный intent: `купить`, `сравнить`, `выбрать`, `узнать`, `связаться`.
- [ ] Одна URL должна закрывать один основной поисковый кластер. Не допускается каннибализация между похожими страницами.
- [ ] Контент создается в первую очередь для пользователей Aerocool, а не для “любого поискового трафика”.
- [ ] Страница должна давать реальную добавленную ценность: собственный опыт, объяснение выбора, сравнение, структурирование, ответ на возражения.
- [ ] Нельзя публиковать тонкий, шаблонный, массово сгенерированный AI-контент без редакторской доработки и фактической проверки.
- [ ] На странице не должно быть фактических ошибок, обещаний без подтверждения, “воды” и keyword stuffing.
- [ ] Нет фиксированного “правильного” объема текста. Пишем столько, сколько нужно для полного закрытия intent.
- [ ] Для проекта `Aerocool` есть рабочий editorial baseline: большие evergreen-статьи обычно `10000+` знаков, а новости, которые должны ранжироваться и поддерживать discovery, обычно `5000+` знаков.
- [ ] Title, H1, description, intro и основной CTA должны быть согласованы друг с другом и обещать одну и ту же пользу.
- [ ] Страница должна быть полезна даже если пользователь зайдет на нее напрямую, без контекста категории или рекламы.
- [ ] После прочтения страницы у пользователя не должно оставаться ощущение “мне нужно снова гуглить то же самое”.

## 3. Универсальный Чек-Лист Для Контента На Любой URL

- [ ] У страницы есть уникальный `title`.
- [ ] У страницы есть уникальный `H1`.
- [ ] У страницы есть уникальный `description`, который описывает реальную пользу и тему страницы.
- [ ] Первые 150-250 слов прямо отвечают на основной запрос.
- [ ] На странице есть четкая структура из подзаголовков под реальные подзапросы пользователя.
- [ ] На странице есть доказательства экспертности: детали, сценарии, спецификации, критерии выбора, сравнения, процесс оценки.
- [ ] На странице есть минимум 3-8 осмысленных внутренних ссылок.
- [ ] У страницы есть сильный следующий шаг: `купить`, `выбрать модель`, `перейти в категорию`, `связаться`, `прочитать сравнение`.
- [ ] Изображения соответствуют теме страницы и не являются случайными иллюстрациями “для красоты”.
- [ ] У изображений осмысленный `alt`, который описывает объект или смысл изображения.
- [ ] На странице нет лишних дат, которые могут запутать Google при определении `datePublished` и `dateModified`.
- [ ] На мобильном экране main content виден сразу и не перекрывается баннерами, попапами или декоративным мусором.

## 4. Как Выигрывать Борьбу За Топ-1 На Практике

- [ ] Целиться нужно не в “просто страницу под ключ”, а в лучший ответ в выдаче по конкретному intent.
- [ ] Для коммерческих запросов товарная страница должна быть лучше страницы конкурента по полноте выбора, спецификациям, возражениям и trust-сигналам.
- [ ] Для информационных запросов статья должна быть лучше по глубине, ясности, опыту, структуре и связке с коммерческими страницами.
- [ ] Для высококонкурентных запросов нужна связка `hub -> category -> product -> article -> faq -> contact`, а не одиночная страница.
- [ ] Успех в топе обеспечивается не только page-level релевантностью, но и site-wide сигналами доверия, качества и тематического покрытия.
- [ ] Страницы с высоким impression и низким CTR нужно регулярно переписывать по `title`, `snippet`, интро и image strategy.
- [ ] Страницы с хорошим CTR, но слабой позицией нужно усиливать по контенту, comparison-блокам, schema и internal links.

## 5. Главная Страница `/`

- [ ] Главная должна объяснять, кто такой Aerocool, что именно продает сайт и для кого он полезен.
- [ ] В первом экране должен быть четкий value proposition, а не только брендовый слоган.
- [ ] Главная должна вести в ключевые money-разделы: каталог, серии, топовые модели, гайды, FAQ, контакты.
- [ ] На главной нужно коротко раскрыть: ассортимент, ключевые преимущества, гарантию, доставку, помощь с подбором.
- [ ] На главной должен быть блок доверия: официальный бренд, поддержка в Украине, сервис, контакты.
- [ ] На главной нужны текстовые ссылки на главные разделы, а не только карточки и кнопки.
- [ ] На главной желательно иметь короткий блок “как выбрать кресло” и ссылки в гайды.
- [ ] На главной нельзя перегружать пользователя многословием; это hub, а не длинная статья.

## 6. Общий Каталог `/products/`

- [ ] Каталог не должен быть просто списком карточек.
- [ ] В верхней части каталога нужен короткий buying guide: кому подходят линейки, чем они отличаются, как подбирать модель.
- [ ] Каталог должен объяснять разницу между игровыми и офисными сценариями использования.
- [ ] На странице должен быть блок “как выбрать по росту, весу, материалу, уровню регулировок”.
- [ ] На странице должны быть ссылки в серии, хиты продаж, статьи и FAQ.
- [ ] Текст каталога должен помогать ранжироваться по более широким коммерческим запросам, не дублируя product pages.
- [ ] Не допускается каннибализация между `/products/` и конкретными series pages.

## 7. Страницы Серий `/products/<series>/`

- [ ] Каждая серия должна иметь свой четкий positioning: кому подходит, чем отличается от других серий, под какой бюджет и сценарии.
- [ ] Серийная страница должна содержать не только grid товаров, но и текст, помогающий выбрать нужную модель.
- [ ] Обязательно описывать ключевые отличия внутри серии: материалы, вентиляция, дизайн, уровень регулировок, посадка.
- [ ] Должен быть comparison-блок серии с соседними сериями Aerocool.
- [ ] Нужно отвечать на вопрос “почему выбрать именно эту серию, а не другую”.
- [ ] Нужно дать внутренние ссылки в конкретные модели серии и в релевантные статьи.
- [ ] Если серия имеет варианты по цветам или материалам, это должно быть отражено в тексте естественно, без переспама.

## 8. Карточки Товаров `/products/<series>/<model>/`

- [ ] У product page есть четкое обещание в `title`: модель + главный use case + коммерческая полезность.
- [ ] В интро есть ответ на вопрос: кому подходит эта модель и почему.
- [ ] В интро нет пустого рекламного пафоса без деталей.
- [ ] Есть блок `кому подходит`.
- [ ] Есть блок `кому не подойдет`.
- [ ] Есть блок с ключевыми преимуществами.
- [ ] Есть полноценная таблица характеристик.
- [ ] Есть блок по эргономике и регулировкам.
- [ ] Есть блок по материалам и ощущению от них в реальном использовании.
- [ ] Есть блок по сценариям: `gaming`, `office`, `home office`, `долгие сессии`, `жаркое помещение`, `высокий рост`, `плотная посадка`.
- [ ] Есть блок по гарантии, возврату, доставке, сборке и сервису.
- [ ] Есть сравнение с 2-3 альтернативами Aerocool.
- [ ] Есть FAQ по предсказуемым objections: рост, вес, ткань/эко-кожа, доставка, сборка, гарантия.
- [ ] Есть CTA на покупку или консультацию.
- [ ] У карточки есть реальная коммерческая завершенность, а не “картинка + 2 абзаца”.
- [ ] На странице должны быть реальные данные: `price`, `sku`, `mpn`, `gtin13`, `availability`, `warranty`, `return_days`.
- [ ] Если данные меняются быстро, обновление контента и schema должно идти синхронно.
- [ ] Для product schema желательно поддерживать не одно изображение, а набор `16:9`, `4:3`, `1:1`, так как Google это рекомендует для merchant experiences.
- [ ] Если модель имеет варианты, нужно либо создать distinct URLs для вариантов, либо корректно реализовать `ProductGroup` / variant markup.
- [ ] Нельзя использовать fake reviews, накрученные ratings и вымышленные факты.

## 9. Страницы Статей `/articles/`

- [ ] Раздел статей должен объяснять, что здесь пользователь найдет: гайды, сравнения, советы по выбору, рабочие сценарии.
- [ ] Раздел статей должен быть связан с деньгами: помогать выбору, а не жить отдельно от каталога.
- [ ] На странице списка статей нужны текстовые вводные под кластер “как выбрать кресло / ergonomics / home office / gaming”.
- [ ] В листинге важно выделять evergreen-гайды и коммерчески полезные comparison-материалы.

## 10. Статьи `/articles/<slug>/`

- [ ] Статья создается под реальный информационный intent, связанный с креслами, эргономикой, рабочим местом или выбором модели.
- [ ] Статья должна усиливать не только брендовые запросы, но и широкие коммерческие кластеры: `игровое кресло`, `офисное кресло`, `компьютерное кресло`, `кресло для работы`, `home office`, если они релевантны теме.
- [ ] Большая evergreen-статья в проекте обычно должна быть не короче `10000` знаков тела на каждую языковую версию.
- [ ] Статья должна содержать original value: собственную рамку оценки, критерии выбора, объяснение trade-offs, практические сценарии.
- [ ] Если статья упоминает “лучшее”, обязательно объяснять критерии и почему.
- [ ] Если статья сравнивает продукты, обязательно описывать, что лучше для какого сценария.
- [ ] Если статья рекомендует товары, должны быть ссылки на конкретные product pages.
- [ ] Если статья использует сторонние данные, она не должна просто пересказывать их без добавления собственной пользы.
- [ ] У статьи должны быть ясные подзапросы в H2/H3, а не искусственные SEO-заголовки.
- [ ] У статьи должна быть авторская или редакторская ответственность.
- [ ] Желательно указывать человека-автора или reviewer, потому что сейчас `Article` schema в проекте указывает только `Organization`.
- [ ] Дата публикации и дата обновления должны быть видимы пользователю и согласованы со structured data.
- [ ] Не менять дату ради “свежести”, если содержание не обновлено по существу.
- [ ] Изображения статьи должны быть не только декоративными, а раскрывать тему.
- [ ] Для article schema также желательно иметь multiple images `16:9`, `4:3`, `1:1`.
- [ ] В конце статьи должен быть следующий шаг: перейти в серию, модель, FAQ или связаться за подбором.

## 11. Раздел Новостей `/news/`

- [ ] News hub должен объяснять, какие именно обновления пользователь здесь найдет: новые модели, поставки, акции, брендовые анонсы.
- [ ] Не стоит превращать раздел новостей в псевдоблог из “новостей ради SEO”.
- [ ] Листинг новостей должен иметь краткую вводную и ссылки на ключевые коммерчески значимые новости.
- [ ] Новости не должны конкурировать с evergreen-статьями за одинаковый intent.

## 12. Новости `/news/<slug>/`

- [ ] Новость публикуется только если есть реальный инфоповод: релиз, запуск серии, поставка, акция, изменение сервиса, партнерство.
- [ ] Новость отвечает на три вопроса: что произошло, почему это важно, что делать дальше.
- [ ] У новости есть четкая дата публикации и, при необходимости, дата обновления.
- [ ] Новость может быть длинной, но не должна быть пустой: каждый дополнительный блок должен раскрывать контекст, сценарии, модели и коммерческую значимость инфоповода.
- [ ] Ranking-supporting новость в проекте обычно должна быть не короче `5000` знаков тела на каждую языковую версию.
- [ ] Новость должна расширять покрытие не только брендовых, но и широких коммерческих запросов, если инфоповод связан с серией, моделью, материалом или сценарием выбора.
- [ ] В новости есть прямые ссылки на соответствующие products, collections или contact page.
- [ ] Если новость time-sensitive, нужно следить, чтобы старые акции не выглядели актуальными спустя месяцы.
- [ ] Если новость больше похожа на гайд или comparison, ее нужно публиковать в `articles`, а не в `news`.

## 13. FAQ `/faq/`

- [ ] FAQ должен отвечать на реальные вопросы пользователей, саппорта и отдела продаж.
- [ ] Все вопросы и ответы должны быть видимы на странице.
- [ ] FAQ должен снимать возражения и помогать конверсии.
- [ ] FAQ полезен как content layer и structured data, но нельзя строить стратегию на ожидании FAQ rich result.
- [ ] По официальной документации Google, FAQ rich results доступны в основном для well-known authoritative health/government sites.
- [ ] На FAQ странице нужно покрывать: подбор, доставка, гарантия, возврат, материалы, сборка, сервис, консультации.
- [ ] FAQ должен вести в товары, статьи и контакты.
- [ ] Если один и тот же FAQ размечается на нескольких страницах, нужно выбрать один канонический экземпляр markup.

## 14. About `/about/`

- [ ] About page должна повышать доверие, а не быть общей “маркетинговой биографией”.
- [ ] Нужно ясно объяснить: кто такой Aerocool в Украине, официальный ли это сайт, какова связь с глобальным брендом, чем подтверждается официальный статус.
- [ ] Нужно показать специализацию: почему бренд компетентен именно в креслах и рабочем комфорте.
- [ ] Нужно раскрыть сервисную модель: поддержка, гарантия, консультация, доставка, работа с клиентами.
- [ ] Желательно усилить страницу реальными доказательствами: команда, процесс подбора, опыт, фото, документы или иные trust elements, если их можно публиковать.
- [ ] About должна быть связана внутренними ссылками с контактами, FAQ, каталогом и guides.

## 15. Contact `/contact/`

- [ ] Contact page должна быть полезной и для людей, и для Google.
- [ ] Контактные данные должны быть полными, точными и единообразными.
- [ ] Телефон, email, адрес, график работы, зона обслуживания и сценарии обращения должны быть явно видимы.
- [ ] Если физического шоурума нет, не создавать ложного локального интента.
- [ ] Если есть локальный адрес и реальное обслуживание, данные на странице должны совпадать с `Organization` / `Business Profile`.
- [ ] Нужны ссылки на каталог, FAQ, статьи и соцсети.
- [ ] Contact page должна отвечать на “по каким вопросам обращаться и что произойдет дальше”.
- [ ] При наличии формы обратной связи нужно указать, сколько обычно занимает ответ.

## 16. Поисковая Страница `/search/`

- [ ] Search page должна хорошо работать для пользователя и быстро отдавать релевантные внутренние результаты.
- [ ] Нужно явно решить, нужна ли ей indexability. Это должно быть осознанное решение, а не случайность.
- [ ] Если страница индексируется, у нее должны быть уникальные title/description и полезный контент.
- [ ] Если страница не должна ранжироваться, это нужно явно зафиксировать технически, а не оставлять на удачу.

## 17. Мультиязычность `uk` + `ru`

- [ ] Для каждой важной страницы должна существовать полная локализованная пара, если страница нужна в обоих языках.
- [ ] Локализация не должна быть формальным переводом; intent и словарь запросов в `uk` и `ru` нужно собирать отдельно.
- [ ] Каждая языковая версия должна ссылаться на себя и на другие версии через `hreflang`.
- [ ] Главный контент должен быть действительно переведен, иначе страницы могут быть восприняты как дубликаты.
- [ ] Title, H1, description, CTA, alt и FAQ переводятся естественно, а не дословно.
- [ ] Внутренние ссылки должны вести на соответствующую языковую версию, а не случайно сбрасывать пользователя на другой язык.
- [ ] Для локализованных страниц нужно следить, чтобы коммерческие факты совпадали: цена, наличие, гарантия, доставка.

## 18. Чек-Лист По Изображениям

- [ ] У каждой важной страницы есть репрезентативное главное изображение.
- [ ] Изображение соответствует реальному содержанию страницы.
- [ ] Изображение доступно для crawl и index.
- [ ] Используются современные форматы и адекватный размер файла.
- [ ] Для ключевых product и article pages желательно иметь изображения в пропорциях `16:9`, `4:3`, `1:1`.
- [ ] Названия файлов осмысленные, а не случайные `img123`.
- [ ] `alt` описывает содержимое изображения, а не повторяет keyword list.
- [ ] Если изображение критично для поиска, оно должно присутствовать на HTML landing page, а не только как CSS background.
- [ ] У product images лучше показывать и изолированный продукт, и реальный контекст использования.

## 19. Чек-Лист По Внутренней Перелинковке

- [ ] Каждая карточка товара ссылается на свою серию.
- [ ] Каждая карточка товара ссылается на 2-4 релевантные статьи.
- [ ] Каждая статья ссылается на 2-5 релевантных товаров и 1-2 серии.
- [ ] FAQ ссылается на каталог, контакт и ключевые статьи.
- [ ] About и Contact ссылаются на каталог, FAQ и trust-critical разделы.
- [ ] Якорные тексты ссылок должны быть описательными, а не “тут”, “подробнее”, “читать”.
- [ ] Основные money pages должны быть доступны по crawlable HTML links.

## 20. Чек-Лист По Заголовкам И Сниппетам

- [ ] У каждой страницы есть описательный и лаконичный `title`.
- [ ] `title` не должен быть чрезмерно длинным или шаблонным.
- [ ] `title` должен отражать реальный content promise страницы.
- [ ] `title`, `H1`, prominent heading и `og:title` должны быть согласованы по смыслу.
- [ ] `description` не должен быть generic; его задача — улучшить качество сниппета и CTR.
- [ ] Нельзя писать description, который обещает то, чего нет на странице.
- [ ] Если CTR страницы низкий, first candidate на переписывание — `title`, intro и image strategy.

## 21. Чек-Лист По Датам

- [ ] На news и article pages дата публикации видна пользователю.
- [ ] Если статья обновлялась по существу, видна и дата обновления.
- [ ] Видимые даты соответствуют значениям structured data.
- [ ] Не используются future dates.
- [ ] На странице нет визуального шума из нерелевантных дат, который может запутать Google.

## 22. Чек-Лист По Structured Data

- [ ] Размечаем только то, что реально присутствует на странице.
- [ ] Не размечаем скрытый, несуществующий или вводящий в заблуждение контент.
- [ ] На товарных страницах поддерживаем точные `price`, `priceCurrency`, `availability`, `sku`.
- [ ] Для товаров по возможности добавляем `material`, `color`, `audience`, variant info, если это доступно в контенте.
- [ ] Для variant-rich товаров рассматриваем внедрение `ProductGroup` и variant markup.
- [ ] На article и news pages указываем автора, headline, image, dates.
- [ ] Breadcrumbs должны соответствовать реальной иерархии.
- [ ] FAQ markup применяем только там, где есть реальный FAQ на странице.
- [ ] Любую новую разметку валидируем через Rich Results Test и Search Console.

## 23. Чек-Лист По Контенту И Доверию

- [ ] Контент должен демонстрировать first-hand expertise или хотя бы явную компетентность и редакторскую ответственность.
- [ ] Нужно создавать страницы, которые пользователь захочет сохранить, переслать или использовать как опору при выборе.
- [ ] Нельзя публиковать материалы на темы, не связанные с core focus сайта, только ради трафика.
- [ ] Нужно иметь сильные trust pages: `About`, `Contact`, `FAQ`, гарантия, доставка, возврат.
- [ ] Желательно иметь страницы авторов или reviewer profiles для guide-heavy разделов.
- [ ] Если есть реальный локальный сервис, нужно синхронизировать сайт с Google Business Profile.
- [ ] Любое claim про официальность, гарантию, сервис и доставку должно подтверждаться на уровне сайта и сервиса.

## 24. Front Matter Чек-Лист По Типам Страниц

- [ ] Для `home`, `about`, `contact`, `collection` и других статичных hub pages обязательно заполнять `title`, `description`, `date`, `lastmod`, `image`, `schema_types`.
- [ ] Для `collection` pages желательно иметь и `summary`, если шаблон использует краткое описание в списках и сниппетах.
- [ ] Для product pages обязательно заполнять `title`, `description`, `summary`, `date`, `lastmod`, `slug`, `categories`, `tags`, `image`, `schema_types`, `price`, `sku`, `availability`, `priceValidUntil`.
- [ ] Для product pages желательно заполнять `mpn`, `gtin13`, `warranty`, `rating.value`, `rating.count`, `return_days`, `shipping_country`, если эти данные доступны.
- [ ] Для article pages обязательно заполнять `title`, `description`, `summary`, `date`, `lastmod`, `slug`, `image`, `schema_types`.
- [ ] Для article pages желательно заполнять `tags`, `categories` или иной тематический классификатор, если он используется в перелинковке.
- [ ] Для news pages обязательно заполнять `title`, `description`, `date`, `lastmod`, `slug`, `image`, `schema_types`.
- [ ] Для news pages желательно заполнять `summary` и `tags`, если это помогает листингам, snippets и internal links.
- [ ] Для FAQ page обязательно заполнять `title`, `description`, `summary`, `date`, `lastmod`, `slug`, `image`, `schema_types`, `faq`.
- [ ] Для FAQ entries все `question` и `answer` должны полностью совпадать с видимым контентом страницы.
- [ ] Для локализованных пар `uk`/`ru` front matter должен быть согласован по фактам, но не обязан быть дословно одинаковым по формулировкам.
- [ ] Нельзя оставлять пустые поля, которые потом попадают в schema, snippets или list templates.

## 25. Анти-Паттерны, Которые Нельзя Делать

- [ ] Не публиковать dozens of thin pages под микроключи без добавленной ценности.
- [ ] Не менять даты ради имитации свежести.
- [ ] Не генерировать много страниц AI-скриптами без редакторской и фактической доработки.
- [ ] Не делать product pages, которые отличаются только цветом, но текстово полностью дублируются без variant strategy.
- [ ] Не создавать новости без реального инфоповода.
- [ ] Не переспамливать titles, H1, alt и anchors повторяющимися ключами.
- [ ] Не использовать fake ratings, fake reviews, fake availability.
- [ ] Не блокировать важные страницы случайным `noindex`, robots или неправильными canonical signals.
- [ ] Не строить стратегию вокруг одного rich result, который Google может не показать.

## 26. Publish Workflow Для Контент-Команды

- [ ] Сначала определяется intent и основная URL.
- [ ] Затем собирается keyword map: primary query, supporting queries, objections, CTA.
- [ ] Затем создается brief с типом страницы и ее ролью в site architecture.
- [ ] Затем пишется черновик под человека, а не под SERP.
- [ ] Затем накладывается SEO-слой: title, H1, description, subheads, internal links, images, front matter.
- [ ] Затем проводится factual QA.
- [ ] Затем проводится SEO QA.
- [ ] Затем страница публикуется и отправляется на recrawl через Search Console.
- [ ] Через 14, 30 и 60 дней страница проходит performance review.

## 27. SEO QA Перед Публикацией

- [ ] URL человекочитаемый и уникальный.
- [ ] `title`, `description`, `H1`, intro, CTA готовы.
- [ ] Front matter заполнен полностью.
- [ ] Schema type соответствует типу страницы.
- [ ] Есть главное изображение.
- [ ] Есть минимум 3 внутренних ссылки.
- [ ] Есть целевой CTA.
- [ ] Видимые факты совпадают с schema и metadata.
- [ ] Страница проверена на мобильном экране.
- [ ] Нет пустых блоков, lorem ipsum, placeholder text, эмодзи-спама и декоративного мусора.

## 28. Что Мониторить После Публикации

- [ ] Indexation в Search Console.
- [ ] Queries, impressions, clicks, CTR, average position.
- [ ] Search appearance по rich results, если они применимы.
- [ ] Страницы с высоким impression и низким CTR.
- [ ] Страницы с ростом показов, но плохой позицией.
- [ ] Страницы с низким вовлечением или слабым переходом на money pages.
- [ ] Новые каннибализации внутри `articles`, `series`, `products`.

## 29. Aerocool-Specific Приоритеты На Следующий Цикл

- [ ] Усилить все product pages до уровня “лучший ответ для покупки”, а не “описание модели”.
- [ ] Добавить comparison-блоки между сериями `WING`, `XTAL`, `SKY` и другими линейками.
- [ ] Превратить `/products/` и series pages в полноценные buying hubs.
- [ ] Усилить статьи first-hand logic, критериями выбора и связкой с товарами.
- [ ] Добавить person-level author/reviewer layer для статей и новостей.
- [ ] Расширить image strategy до `16:9`, `4:3`, `1:1` на топовых страницах.
- [ ] При необходимости внедрить `ProductGroup` для вариантов.
- [ ] Регулярно обновлять FAQ по реальным вопросам продаж и поддержки.
