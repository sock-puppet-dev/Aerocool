# План Inline-Изображений Для Статей И Новостей — 2026-06-14

Актуально на 2026-06-14.

Этот документ фиксирует ручной анализ всех материалов `content/articles` и `content/news`: какие дополнительные изображения нужны внутри тела каждой статьи и каждой новости, где их ставить и какие технические параметры использовать.

## 1. Источники И Принципы

Внешние источники, с которыми сверялся стандарт:

- Google Search Central, Image SEO Best Practices: https://developers.google.com/search/docs/appearance/google-images
- Google Search Central, Article structured data: https://developers.google.com/search/docs/appearance/structured-data/article
- web.dev, Learn Images: https://web.dev/learn/images
- web.dev, Fetch Priority API: https://web.dev/articles/fetch-priority

Практический вывод для проекта:

- изображение должно быть в HTML через `<img>` / `<picture>`, а не только CSS background;
- `alt`, filename, heading и соседний текст должны описывать одну и ту же сущность;
- для `Article` / `NewsArticle` нужен основной набор `01-front.webp` + crops **16:9**, **4:3**, **1:1**;
- LCP-изображение может получать высокий приоритет, но inline-изображения в теле статьи должны быть lazy;
- top-1 в SERP нельзя гарантировать только картинками, но качественные изображения повышают релевантность, кликабельность, понятность и право страницы участвовать в image/search surfaces.

## 2. Единый Технический Стандарт Inline-Изображений

Для всех рекомендаций ниже:

- формат: **WebP**;
- размер по умолчанию: **1200x800 px**;
- допустимый широкий вариант: **1200x675 px** для сравнительных сцен;
- целевой вес: **120-280 КБ**, максимум **350 КБ**;
- имя: `02-<topic>.webp`, `03-<topic>.webp`, далее по порядку;
- хранение: рядом с `index.md` и `index.ru.md` в page bundle;
- вывод: через `seo-image`;
- loading: lazy;
- decoding: async;
- без `preload=true` и без `fetchpriority="high"`;
- без текста, цифр, UI labels и фальшивых характеристик внутри изображения;
- `alt` локализовать отдельно для `index.md` и `index.ru.md`;
- если нужны точные характеристики, они должны быть в HTML-тексте, а не только на картинке.

## 3. Приоритеты

P1:

- добавить inline-изображения в evergreen articles, где есть сравнения, механизмы, материалы или пошаговая настройка;
- добавить хотя бы одно inline-изображение в каждую новость, где сейчас после обложки идет длинный текст без дополнительного визуального объяснения.

P2:

- сделать factual material close-ups и mechanism close-ups на основе официальных product assets;
- добавить SVG/WebP схемы для `Synchronous Tilt`, `SYNC4/SYNC5`, `7D/8D/11D`, если схема не содержит недоказанных спецификаций.

P3:

- после появления реальных product gallery assets заменить часть AI-editorial inline scenes на factual product detail images.

## 6. Правила Вставки В Markdown

Пример для украинской версии:

```go-html-template
{{< seo-image
  src="02-series-choice-map.webp"
  alt="Візуальна схема вибору між серіями Aerocool SKY, WING і XTAL"
  loading="lazy"
>}}
```

Для русской версии:

```go-html-template
{{< seo-image
  src="02-series-choice-map.webp"
  alt="Визуальная схема выбора между сериями Aerocool SKY, WING и XTAL"
  loading="lazy"
>}}
```

Правила:

- один и тот же файл можно использовать в `index.md` и `index.ru.md`, если внутри изображения нет текста;
- `alt` всегда писать на языке страницы;
- картинку вставлять после релевантного `##`, а не в случайном месте;
- не вставлять inline-изображение сразу после обложки, если следующий блок еще не раскрыл тему;
- не ставить больше двух изображений подряд без текста между ними;
- не превращать HTML-таблицы, FAQ и сравнительные списки в bitmap.

## 7. Оценка Текущего Состояния

Сейчас у всех article/news bundles есть primary cover и schema crops. Это сильная база.

Оценка inline-image слоя: **6.5 / 10**.

Почему не выше:

- внутри тела материалов почти везде есть только первое изображение-обложка;
- статьи с механизмами, материалами и сравнением серий не имеют дополнительных visual explainers;
- новости про запуск серий и моделей не показывают второй уровень фактического объяснения;
- часть тем требует factual close-ups, которых нельзя корректно заменить AI-иллюстрацией без официальных product assets.

После добавления рекомендованных inline-изображений ожидаемая оценка: **8.8-9.2 / 10** при условии, что изображения будут релевантными, легкими, с локализованным `alt` и без фальшивых характеристик.
