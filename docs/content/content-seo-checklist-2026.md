# SEO-Чек-Лист Контента Aerocool 2026

Актуально на `2026-05-07`.

Этот документ — единый редакционный и SEO-регламент для всего проекта `Aerocool`. Он адаптирован под текущую структуру сайта на `Hugo` с разделами `home`, `products`, `articles`, `news`, `faq`, `about`, `contact`, `search`, а также под двуязычную архитектуру `uk` + `ru`.

Важно: ни один чек-лист не гарантирует `топ-1` в Google. По официальным рекомендациям Google, даже если страница соблюдает Search Essentials и лучшие практики, это не означает гарантированный обход, индексирование или ранжирование. Цель этого документа — максимально повысить вероятность сильного ранжирования, высокого CTR и коммерческой полезности.

Текущий практический порядок работ после обновления документации описан в [2026-05-07-documentation-refresh-and-project-action-plan.md](/Users/stadnyk/MEGA/Aerocool/docs/audits/2026-05-07-documentation-refresh-and-project-action-plan.md).

Как пользоваться новичку: перед публикацией страницы бери нужный раздел этого чек-листа, проходи пункты сверху вниз и отмечай только то, что действительно выполнено в тексте, front matter, изображениях, ссылках и schema.org. Если пункт непонятен, сначала смотри [README.md](/Users/stadnyk/MEGA/Aerocool/README.md), [front-matter.md](/Users/stadnyk/MEGA/Aerocool/docs/content/front-matter-reference.md) и [helpers.md](/Users/stadnyk/MEGA/Aerocool/docs/architecture/hugo-template-helpers.md).

## 2. Неподвижные Правила Для Любой Страницы

- [ ] У каждой URL должен быть один главный поисковый интент: `купить`, `сравнить`, `выбрать`, `узнать`, `связаться`.
- [ ] Одна URL должна закрывать один основной поисковый кластер. Не допускается каннибализация между похожими страницами.
- [ ] Контент создается в первую очередь для пользователей Aerocool, а не для “любого поискового трафика”.
- [ ] Страница должна давать реальную добавленную ценность: собственный опыт, объяснение выбора, сравнение, структурирование, ответ на возражения.
- [ ] Нельзя публиковать тонкий, шаблонный, массово сгенерированный AI-контент без редакторской доработки и фактической проверки.
- [ ] На странице не должно быть фактических ошибок, обещаний без подтверждения, “воды” и переспама ключевыми словами.
- [ ] Нет фиксированного “правильного” объема текста. Пишем столько, сколько нужно для полного закрытия интента.
- [ ] Для проекта `Aerocool` есть рабочие редакционные ориентиры по объему: статьи обычно `10000+` знаков, новости обычно `5000+` знаков, товарные страницы обычно `6000+` знаков, страницы серий обычно `6000+` знаков, хабы `/products/`, `/articles/`, `/news/` обычно `7000+` знаков, страница `/about/` обычно `10000+` знаков основного текста на каждую языковую версию.
- [ ] Расширение до редакционного ориентира должно быть смысловым: добавляем сценарии выбора, сравнения, практические проверки, FAQ, следующий шаг и внутренние ссылки, а не абзацы ради объема.
- [ ] Контент должен быть целостным, а не узко-keyword-страницей: закрывать сущность, связанные подтемы, атрибуты, сценарии и последующие вопросы пользователя.
- [ ] Для важной сущности должна быть понятная `entity home`-страница: URL, который лучше всего определяет бренд, серию, модель, материал, механизм или сценарий. Внешние источники могут уточнять сущность, но не должны заменять собственное определение на сайте.
- [ ] После расширения или литературной обработки `uk` и `ru` версии должны оставаться синхронными по фактам, структуре аргумента и коммерческому интенту, но формулировки могут быть естественно локализованы.
- [ ] `Title`, `H1`, `description`, интро и основной CTA должны быть согласованы друг с другом и обещать одну и ту же пользу.
- [ ] Для большинства страниц в текущем проекте видимый `H1` принадлежит шаблонному слою и рендерится по правилу `h1 | default title`; исключение сейчас — главная, где `H1` живет в `layouts/_shortcodes/home-hero.html`. В теле markdown не должно быть `# H1`.
- [ ] Фирменные и технические термины пишутся единообразно во всех языковых версиях, front matter, видимом тексте и внутренних ссылках. Для механизма наклона стандарт проекта — `Synchronous Tilt`; нижнерегистровый вариант не использовать.
- [ ] В markdown-контенте все inline-code фрагменты оформляются жирным code-стилем: ``**`11D`**``, ``**`SYNC5 multi-adjustable`**``, ``**`офисное кресло`**``. Это правило применяется к видимому контенту в `content/**/*.md`, но не к Hugo/JS-коду в `layouts/`.
- [ ] Контент проектируется компонентами, а не только “страницей целиком”: интро, короткий ответ, блок выбора, характеристики, сравнение, FAQ, доверительный блок, CTA и внутренние ссылки должны быть повторяемыми паттернами.
- [ ] Для одинаковых типов страниц использовать одинаковую структуру, терминологию и метаданные, чтобы человек, Hugo-шаблоны, schema.org и AI-системы видели предсказуемые связи.
- [ ] Формулировки должны быть семантическими, а не плоскими: вместо расплывчатого “узнайте больше о наших решениях” называть конкретную сущность, сценарий и связь, например “узнайте, как серия Aerocool SKY подходит для home office и долгой работы за компьютером”.
- [ ] Контент должен быть пригоден для переиспользования в разных поверхностях: страница сайта, schema.org, сниппет, AI summary, чатовый ответ или будущий агентский интерфейс.
- [ ] Фактические утверждения должны быть непротиворечивыми между видимым текстом, front matter, JSON-LD, FAQ, товарными характеристиками и локализованными версиями.
- [ ] Страница должна быть полезна даже если пользователь зайдет на нее напрямую, без контекста категории или рекламы.
- [ ] После прочтения страницы у пользователя не должно оставаться ощущение “мне нужно снова гуглить то же самое”.

## 3. Универсальный Чек-Лист Для Контента На Любой URL

- [ ] У страницы есть уникальный `title`.
- [ ] У страницы есть уникальный `H1`.
- [ ] У страницы есть уникальный `description`, который описывает реальную пользу и тему страницы.
- [ ] Если нужен отдельный видимый заголовок, он задается через front matter `h1`; если он не нужен, layout берет `title`. Исключение на текущем этапе — главная страница, где видимый `H1` задается в `layouts/_shortcodes/home-hero.html`.
- [ ] Первые 150-250 слов прямо отвечают на основной запрос.
- [ ] На странице есть четкая структура из подзаголовков под реальные подзапросы пользователя.
- [ ] На странице есть доказательства экспертности: детали, сценарии, спецификации, критерии выбора, сравнения, процесс оценки.
- [ ] На странице есть минимум 3-8 осмысленных внутренних ссылок.
- [ ] У страницы есть сильный следующий шаг: `купить`, `выбрать модель`, `перейти в категорию`, `связаться`, `прочитать сравнение`.
- [ ] Изображения соответствуют теме страницы и не являются случайными иллюстрациями “для красоты”.
- [ ] У изображений осмысленный `alt`, который описывает объект или смысл изображения.
- [ ] На странице нет лишних дат, которые могут запутать Google при определении `datePublished` и `dateModified`.
- [ ] На мобильном экране main content виден сразу и не перекрывается баннерами, попапами или декоративным мусором.

## 4. Как Выигрывать Борьбу За Топ-1 На Практике

- [ ] Целиться нужно не в “просто страницу под ключ”, а в лучший ответ в выдаче по конкретному интенту.
- [ ] Для коммерческих запросов товарная страница должна быть лучше страницы конкурента по полноте выбора, спецификациям, работе с возражениями и сигналам доверия.
- [ ] Для информационных запросов статья должна быть лучше по глубине, ясности, опыту, структуре и связке с коммерческими страницами.
- [ ] Для высококонкурентных запросов нужна связка `хаб -> категория -> товар -> статья -> faq -> контакты`, а не одиночная страница.
- [ ] Успех в топе обеспечивается не только релевантностью отдельной страницы, но и сигналами доверия, качества и тематического покрытия на уровне всего сайта.
- [ ] Страницы с высокими показами и низким CTR нужно регулярно переписывать по `title`, сниппету, интро и стратегии изображений.
- [ ] Страницы с хорошим CTR, но слабой позицией нужно усиливать по контенту, блокам сравнения, schema и внутренним ссылкам.

## 5. Главная Страница `/`

- [ ] Главная должна объяснять, кто такой Aerocool, что именно продает сайт и для кого он полезен.
- [ ] В первом экране должно быть четкое ценностное предложение, а не только брендовый слоган.
- [ ] Главная должна вести в ключевые money-разделы: каталог, серии, топовые модели, гайды, FAQ, контакты.
- [ ] На главной нужно коротко раскрыть: ассортимент, ключевые преимущества, гарантию, доставку, помощь с подбором.
- [ ] На главной должен быть блок доверия: официальный бренд, поддержка в Украине, сервис, контакты.
- [ ] На главной нужны текстовые ссылки на главные разделы, а не только карточки и кнопки.
- [ ] На главной желательно иметь короткий блок “как выбрать кресло” и ссылки в гайды.
- [ ] На главной нельзя перегружать пользователя многословием; это хаб, а не длинная статья.
- [ ] Hero-изображение главной должно быть локальным, а не внешней demo-картинкой; для первого экрана использовать `relURL`, `loading="eager"` и `fetchpriority="high"`.

## 6. Общий Каталог `/products/`

- [ ] Каталог не должен быть просто списком карточек.
- [ ] Общий каталог `/products/` в проекте обычно должен быть не короче `7000` знаков основного текста на каждую языковую версию.
- [ ] В верхней части каталога нужен короткий гид по выбору: кому подходят линейки, чем они отличаются и как подбирать модель.
- [ ] Каталог должен объяснять разницу между игровыми и офисными сценариями использования.
- [ ] На странице должен быть блок “как выбрать по росту, весу, материалу, уровню регулировок”.
- [ ] На странице должны быть ссылки в серии, хиты продаж, статьи и FAQ.
- [ ] Текст каталога должен помогать ранжироваться по более широким коммерческим запросам, не дублируя товарные страницы.
- [ ] Не допускается каннибализация между `/products/` и конкретными страницами серий.

## 7. Страницы Серий `/products/<series>/`

- [ ] Каждая серия должна иметь четкое позиционирование: кому подходит, чем отличается от других серий, под какой бюджет и сценарии рассчитана.
- [ ] Страница серии в проекте обычно должна быть не короче `6000` знаков основного текста на каждую языковую версию.
- [ ] Страница серии должна содержать не только сетку товаров, но и текст, помогающий выбрать нужную модель.
- [ ] Обязательно описывать ключевые отличия внутри серии: материалы, вентиляция, дизайн, уровень регулировок, посадка.
- [ ] Должен быть блок сравнения серии с соседними сериями Aerocool.
- [ ] Нужно отвечать на вопрос “почему выбрать именно эту серию, а не другую”.
- [ ] Нужно дать внутренние ссылки в конкретные модели серии и в релевантные статьи.
- [ ] Если серия имеет варианты по цветам или материалам, это должно быть отражено в тексте естественно, без переспама.

## 8. Карточки Товаров `/products/<series>/<model>/`

- [ ] У товарной страницы должно быть четкое обещание в `title`: модель + основной сценарий использования + коммерческая польза.
- [ ] Товарная страница в проекте обычно должна быть не короче `6000` знаков основного текста на каждую языковую версию.
- [ ] В интро есть ответ на вопрос: кому подходит эта модель и почему.
- [ ] В интро нет пустого рекламного пафоса без деталей.
- [ ] Есть блок `кому подходит`.
- [ ] Есть блок `кому не подойдет`.
- [ ] Есть блок с ключевыми преимуществами.
- [ ] Есть полноценная таблица характеристик.
- [ ] Есть блок по эргономике и регулировкам.
- [ ] Есть блок по материалам и ощущению от них в реальном использовании.
- [ ] Есть блок по сценариям: `gaming`, `office`, `home office`, `долгие сессии`, `жаркое помещение`, `высокий рост`, `плотная посадка`.
- [ ] Есть блок по гарантии, возврату, доставке, сборке и сервису.
- [ ] Есть сравнение с 2-3 альтернативами Aerocool.
- [ ] Есть FAQ по предсказуемым возражениям: рост, вес, ткань/эко-кожа, доставка, сборка, гарантия.
- [ ] Есть CTA на покупку или консультацию.
- [ ] У карточки есть реальная коммерческая завершенность, а не “картинка + 2 абзаца”.
- [ ] На странице должны быть реальные данные: `price`, `sku`, `mpn`, `gtin13`, `availability`, `warranty`, `return_days`.
- [ ] Цена, наличие, гарантия, доставка, возврат, рейтинг и количество отзывов должны быть видимыми на странице, если эти же данные выводятся в `Product` JSON-LD.
- [ ] Merchant-условия из `Product` JSON-LD должны совпадать с `/faq/`: бесплатная доставка по Украине `Новой Почтой`, передача в отправку `0-1 день`, транзит `1-3 дня`, возврат `14 дней`, бесплатный возврат и оплата наличными/картой.
- [ ] Если данные меняются быстро, обновление контента и schema должно идти синхронно.
- [ ] Для product image в текущем проекте использовать связку `image + cover.image + seo-image`, а не только один shortcode или только одно поле front matter.
- [ ] Для product schema желательно поддерживать не одно изображение, а набор `16:9`, `4:3`, `1:1`, так как Google рекомендует это для товарных выдач.
- [ ] Если модель имеет варианты, нужно либо создать отдельные URL для вариантов, либо корректно реализовать `ProductGroup` и разметку вариантов.
- [ ] Нельзя использовать фальшивые отзывы, накрученные рейтинги и вымышленные факты.

## 9. Страницы Статей `/articles/`

- [ ] Раздел статей должен объяснять, что здесь пользователь найдет: гайды, сравнения, советы по выбору, рабочие сценарии.
- [ ] Хаб статей `/articles/` в проекте обычно должен быть не короче `7000` знаков основного текста на каждую языковую версию.
- [ ] Раздел статей должен быть связан с деньгами: помогать выбору, а не жить отдельно от каталога.
- [ ] На странице списка статей нужны текстовые вводные под кластер “как выбрать кресло / эргономика / home office / gaming”.
- [ ] В листинге важно выделять постоянно актуальные гайды и коммерчески полезные статьи со сравнениями.

## 10. Статьи `/articles/<slug>/`

- [ ] Статья создается под реальный информационный интент, связанный с креслами, эргономикой, рабочим местом или выбором модели.
- [ ] Статья должна усиливать не только брендовые запросы, но и широкие коммерческие кластеры: `игровое кресло`, `офисное кресло`, `компьютерное кресло`, `кресло для работы`, `home office`, если они релевантны теме.
- [ ] Большая постоянно актуальная статья в проекте обычно должна быть не короче `10000` знаков основного текста на каждую языковую версию.
- [ ] Статья должна содержать оригинальную ценность: собственную рамку оценки, критерии выбора, объяснение компромиссов и практические сценарии.
- [ ] Литературная обработка статьи должна улучшать ясность, ритм и пользу для выбора, сохраняя SEO-интент и не превращая текст в рекламный монолог.
- [ ] Если статья упоминает “лучшее”, обязательно объяснять критерии и почему.
- [ ] Если статья сравнивает продукты, обязательно описывать, что лучше для какого сценария.
- [ ] Если статья рекомендует товары, должны быть ссылки на конкретные товарные страницы.
- [ ] Если статья использует сторонние данные, она не должна просто пересказывать их без добавления собственной пользы.
- [ ] У статьи должны быть ясные подзапросы в H2/H3, а не искусственные SEO-заголовки.
- [ ] У статьи должна быть авторская или редакторская ответственность.
- [ ] В текущем проекте видимая редакционная ответственность для статей и новостей выводится через [layouts/_partials/editorial-note.html](/Users/stadnyk/MEGA/Aerocool/layouts/_partials/editorial-note.html).
- [ ] Желательно указывать автора или редактора, потому что сейчас разметка `Article` в проекте указывает только `Organization`.
- [ ] Дата публикации и дата обновления должны быть видимы пользователю и согласованы со structured data.
- [ ] Не менять дату ради “свежести”, если содержание не обновлено по существу.
- [ ] Изображения статьи должны быть не только декоративными, а раскрывать тему.
- [ ] Для разметки `Article` также желательно иметь несколько изображений в пропорциях `16:9`, `4:3`, `1:1`.
- [ ] В конце статьи должен быть следующий шаг: перейти в серию, модель, FAQ или связаться за подбором.

## 11. Раздел Новостей `/news/`

- [ ] Раздел новостей должен объяснять, какие именно обновления пользователь здесь найдет: новые модели, поставки, акции, брендовые анонсы.

- [ ] Не стоит превращать раздел новостей в псевдоблог из “новостей ради SEO”.
- [ ] Листинг новостей должен иметь краткую вводную и ссылки на ключевые коммерчески значимые новости.
- [ ] Новости не должны конкурировать с постоянно актуальными статьями за одинаковый интент.

## 12. Новости `/news/<slug>/`

- [ ] Новость публикуется только если есть реальный инфоповод: релиз, запуск серии, поставка, акция, изменение сервиса, партнерство.
- [ ] Новость отвечает на три вопроса: что произошло, почему это важно, что делать дальше.
- [ ] У новости есть четкая дата публикации и, при необходимости, дата обновления.
- [ ] Новость может быть длинной, но не должна быть пустой: каждый дополнительный блок должен раскрывать контекст, сценарии, модели и коммерческую значимость инфоповода.
- [ ] Новость, поддерживающая ранжирование, в проекте обычно должна быть не короче `5000` знаков тела на каждую языковую версию.
- [ ] Литературная обработка новости должна сохранять новостной повод в центре: что произошло, почему это важно, кому полезно и куда перейти дальше.
- [ ] Новость должна расширять покрытие не только брендовых, но и широких коммерческих запросов, если инфоповод связан с серией, моделью, материалом или сценарием выбора.
- [ ] В новости есть прямые ссылки на соответствующие товары, коллекции или страницу контактов.
- [ ] Если новость чувствительна ко времени, нужно следить, чтобы старые акции не выглядели актуальными спустя месяцы.
- [ ] Если новость больше похожа на гайд или сравнение, ее нужно публиковать в `articles`, а не в `news`.

## 13. FAQ `/faq/`

- [ ] FAQ должен отвечать на реальные вопросы пользователей, саппорта и отдела продаж.
- [ ] Все вопросы и ответы должны быть видимы на странице.

- [ ] FAQ должен снимать возражения и помогать конверсии.
- [ ] FAQ полезен как контентный слой и structured data, но нельзя строить стратегию на ожидании FAQ rich result.
- [ ] По официальной документации Google, FAQ rich results доступны в основном для well-known authoritative health/government sites.
- [ ] Для текущего проекта FAQ в первую очередь должен покрывать: выбор серии, материалы, механики и регулировки, гарантию, доставку, оплату, возврат, сборку и консультации.
- [ ] Для Aerocool особенно важны вопросы про `SKY`, `WING`, `XTAL`, `Racer`, `Loft Air`, `Mesh`, `Synchronous Tilt`, `7D`, `8D`, `11D`, подлокотники и сценарии `work`, `home office`, `gaming`.
- [ ] Возврат, сборка, способы оплаты, рассрочка, параметры роста/веса и другие сервисные детали включаются в FAQ только если они явно подтверждены на уровне сайта и сервиса.
- [ ] FAQ должен вести в товары, статьи и контакты.
- [ ] Если один и тот же FAQ размечается на нескольких страницах, нужно выбрать один канонический экземпляр markup.
- [ ] Длинный FAQ-хаб на `30+` вопросов допустим, если каждая запись остается полезной, недублирующей и подтвержденной контентом сайта.
- [ ] FAQ не должен звучать как шаблонная AI-подборка: ответы должны быть литературно отредактированы, естественны по тону и без канцелярской машинности.

## 14. Страница О Бренде `/about/`

- [ ] Страница о бренде должна повышать доверие, а не быть общей “маркетинговой биографией”.

- [ ] Желательно усилить страницу реальными доказательствами: команда, процесс подбора, опыт, фото, документы или иные элементы доверия, если их можно публиковать.
- [ ] About должна быть связана внутренними ссылками с контактами, FAQ, каталогом и guides.

## 15. Страница Контактов `/contact/`

- [ ] Страница контактов должна быть полезной и для людей, и для Google.

## 16. Поисковая Страница `/search/`

- [ ] Страница поиска может иметь собственные `title`, `description`, `summary`, `layout` и `placeholder` для UX, но не является целевой страницей для ранжирования.

## 17. Мультиязычность `uk` + `ru`

- [ ] Для каждой важной страницы должна существовать полная локализованная пара, если страница нужна в обоих языках.
- [ ] Локализация не должна быть формальным переводом; интент и словарь запросов в `uk` и `ru` нужно собирать отдельно.
- [ ] Каждая языковая версия должна ссылаться на себя и на другие версии через `hreflang`.
- [ ] Главный контент должен быть действительно переведен, иначе страницы могут быть восприняты как дубликаты.
- [ ] Title, H1, description, CTA, alt и FAQ переводятся естественно, а не дословно.
- [ ] Внутренние ссылки должны вести на соответствующую языковую версию, а не случайно сбрасывать пользователя на другой язык.
- [ ] Для локализованных страниц нужно следить, чтобы коммерческие факты совпадали: цена, наличие, гарантия, доставка.

## 18. Чек-Лист По Изображениям

- [ ] У каждой важной страницы есть репрезентативное главное изображение.
- [ ] Изображение соответствует реальному содержанию страницы.
- [ ] Изображение доступно для crawl и index.
- [ ] Используются современные форматы и адекватный размер файла.
- [ ] Для ключевых товарных страниц и статей желательно иметь изображения в пропорциях `16:9`, `4:3`, `1:1`.
- [ ] Названия файлов осмысленные, а не случайные `img123`.
- [ ] `alt` описывает содержимое изображения, а не повторяет keyword list.
- [ ] Если изображение критично для поиска, оно должно присутствовать на HTML-странице, а не только как CSS-фон.
- [ ] У product images лучше показывать и изолированный продукт, и реальный контекст использования.

## 19. Чек-Лист По Внутренней Перелинковке

- [ ] Каждая карточка товара ссылается на свою серию.
- [ ] Каждая карточка товара ссылается на 2-4 релевантные статьи.
- [ ] Каждая статья ссылается на 2-5 релевантных товаров и 1-2 серии.
- [ ] FAQ ссылается на каталог, контакт и ключевые статьи.
- [ ] About и Contact ссылаются на каталог, FAQ и критически важные доверительные разделы.
- [ ] Якорные тексты ссылок должны быть описательными, а не “тут”, “подробнее”, “читать”.
- [ ] Основные коммерческие страницы должны быть доступны по crawlable HTML-ссылкам.

## 20. Чек-Лист По Заголовкам И Сниппетам

- [ ] У каждой страницы есть описательный и лаконичный `title`.
- [ ] `title` не должен быть чрезмерно длинным или шаблонным.
- [ ] `title` должен отражать реальное обещание страницы.
- [ ] `title`, `H1`, основной видимый заголовок и `og:title` должны быть согласованы по смыслу.
- [ ] `description` не должен быть шаблонным; его задача — улучшить качество сниппета и CTR.
- [ ] Нельзя писать `description`, который обещает то, чего нет на странице.
- [ ] Если CTR страницы низкий, в первую очередь пересматриваются `title`, интро и стратегия изображений.

## 21. Чек-Лист По Датам

- [ ] На новостях и статьях дата публикации должна быть видна пользователю.
- [ ] Если статья обновлялась по существу, видна и дата обновления.
- [ ] Видимые даты соответствуют значениям structured data.
- [ ] Не используются future dates.
- [ ] На странице нет визуального шума из нерелевантных дат, который может запутать Google.

## 22. Чек-Лист По Структурированным Данным

- [ ] Размечаем только то, что реально присутствует на странице.
- [ ] Не размечаем скрытый, несуществующий или вводящий в заблуждение контент.
- [ ] На товарных страницах поддерживаем точные `price`, `priceCurrency`, `availability`, `sku`.
- [ ] Для Google rich results проверяем не только Schema Validator, но и совпадение JSON-LD с видимым контентом страницы.
- [ ] Для товаров по возможности добавляем `material`, `color`, `audience` и данные о вариантах, если это доступно в контенте.
- [ ] Для товаров с большим числом вариантов рассматриваем внедрение `ProductGroup` и разметки вариантов.
- [ ] На статьях и новостях указываем автора, `headline`, `image` и даты.
- [ ] Breadcrumbs должны соответствовать реальной иерархии.
- [ ] FAQ markup применяем только там, где есть реальный FAQ на странице.
- [ ] Любую новую разметку валидируем через Rich Results Test и Search Console.

## 23. Чек-Лист По Контенту И Доверию

- [ ] Контент должен демонстрировать личный практический опыт или хотя бы явную компетентность и редакторскую ответственность.
- [ ] Нужно создавать страницы, которые пользователь захочет сохранить, переслать или использовать как опору при выборе.
- [ ] Нельзя публиковать материалы на темы, не связанные с основной специализацией сайта, только ради трафика.
- [ ] Нужно иметь сильные доверительные страницы: `About`, `Contact`, `FAQ`, гарантия, доставка, возврат.
- [ ] Желательно иметь страницы авторов или профили редакторов и ревьюеров для разделов, где много гайдов и сравнений.
- [ ] Если есть реальный локальный сервис, нужно синхронизировать сайт с Google Business Profile.
- [ ] Любое утверждение про официальность, гарантию, сервис и доставку должно подтверждаться на уровне сайта и сервиса.

## 24. Чек-Лист Полей Метаданных По Типам Страниц

- [ ] Для `home`, `about`, `contact`, `collection` и других статичных хаб-страниц обязательно заполнять `title`, `description`, `date`, `lastmod`, `image`, `schema_types`.
- [ ] Поле `h1` в метаданных страницы необязательно и используется только тогда, когда видимый заголовок должен отличаться от SEO `title`.
- [ ] Для `collection`-страниц желательно иметь и `summary`, если шаблон использует краткое описание в списках и сниппетах.
- [ ] Для товарных страниц обязательно заполнять `title`, `description`, `summary`, `date`, `lastmod`, `slug`, `categories`, `tags`, `image`, `schema_types`, `price`, `sku`, `availability`, `priceValidUntil`.
- [ ] Для товарных страниц желательно заполнять `mpn`, `gtin13`, `warranty`, `rating.value`, `rating.count`, `return_days`, `shipping_country`, если эти данные доступны.
- [ ] Если меняются доставка, возврат или способы оплаты, обновлять `/faq/`, `Product` JSON-LD и видимый коммерческий блок товарных страниц синхронно.
- [ ] Для статей обязательно заполнять `title`, `description`, `summary`, `date`, `lastmod`, `slug`, `image`, `schema_types`.
- [ ] Для статей желательно заполнять `tags`, `categories` или иной тематический классификатор, если он используется в перелинковке.
- [ ] Для новостей обязательно заполнять `title`, `description`, `date`, `lastmod`, `slug`, `image`, `schema_types`.
- [ ] Для новостей желательно заполнять `summary` и `tags`, если это помогает листингам, сниппетам и внутренним ссылкам.
- [ ] Для FAQ-страницы обязательно заполнять `title`, `description`, `summary`, `date`, `lastmod`, `slug`, `image`, `schema_types`, `faq`.
- [ ] Для FAQ-записей все `question` и `answer` должны полностью совпадать с видимым контентом страницы.
- [ ] Для локализованных пар `uk`/`ru` метаданные страницы должны быть согласованы по фактам, но не обязаны быть дословно одинаковыми по формулировкам.
- [ ] Нельзя оставлять пустые поля, которые потом попадают в schema.org-разметку, сниппеты или шаблоны списков.
- [ ] `404` и служебные alias-страницы не являются контентными посадочными страницами и не описываются через метаданные в `content/`; они обслуживаются шаблонным слоем.

## 25. Анти-Паттерны, Которые Нельзя Делать

- [ ] Не публиковать десятки тонких страниц под микроключи без добавленной ценности.
- [ ] Не менять даты ради имитации свежести.
- [ ] Не генерировать много страниц AI-скриптами без редакторской и фактической доработки.
- [ ] Не делать товарные страницы, которые отличаются только цветом, но текстово полностью дублируются без стратегии вариантов.
- [ ] Не создавать новости без реального инфоповода.
- [ ] Не переспамливать `title`, `H1`, `alt` и анкоры повторяющимися ключами.
- [ ] Не использовать фальшивые рейтинги, отзывы и наличие.
- [ ] Не блокировать важные страницы случайным `noindex`, robots или неправильными canonical signals.
- [ ] Не строить стратегию вокруг одного rich result, который Google может не показать.

## 26. Публикационный Процесс Для Контент-Команды

- [ ] Сначала определяется интент и основная URL.
- [ ] Затем собирается карта ключевых слов: основной запрос, поддерживающие запросы, возражения, CTA.
- [ ] Затем создается бриф с типом страницы и ее ролью в архитектуре сайта.
- [ ] Затем определяется компонентная структура страницы: какие блоки нужны человеку, какие блоки будут источником для schema.org, какие фрагменты могут быть переиспользованы в AI-ответах.
- [ ] Затем фиксируются главные сущности страницы и связи между ними: бренд, серия, модель, материал, механизм, сценарий, доставка, гарантия, возврат.
- [ ] Затем определяется `entity home`: является ли эта страница главным определением сущности или должна ссылаться на другую каноническую страницу.
- [ ] Затем собираются связанные подтемы и follow-up вопросы, которые могут появиться в AI snapshot или разговорном поиске. Не ограничиваться только Google `People Also Ask`.
- [ ] Затем пишется черновик для человека, а не для SERP.
- [ ] Затем накладывается SEO-слой: `title`, необязательный `h1`, `description`, подзаголовки, внутренние ссылки, изображения и метаданные страницы.
- [ ] Затем проверяется семантическая ясность текста: в ключевых местах названы конкретные сущности и отношения, а не общие фразы без контекста.
- [ ] Затем проводится фактическая проверка.
- [ ] Затем проводится SEO-проверка.
- [ ] Затем страница публикуется и отправляется на повторный обход через Search Console.
- [ ] Через 14, 30 и 60 дней страница проходит performance review.

## 27. SEO-Проверка Перед Публикацией

- [ ] URL человекочитаемый и уникальный.
- [ ] `title`, необязательный `h1`, `description`, интро и CTA готовы.
- [ ] Метаданные страницы заполнены полностью.
- [ ] Тип schema.org-разметки соответствует типу страницы.
- [ ] Есть главное изображение.
- [ ] Есть минимум 3 внутренних ссылки.
- [ ] Есть целевой CTA.
- [ ] Видимые факты совпадают со schema.org-разметкой и метаданными.
- [ ] Понятно, какая сущность является главной для страницы и где находится ее `entity home`.
- [ ] Страница покрывает не только основной запрос, но и связанные подтемы, атрибуты, сравнения и вероятные follow-up вопросы.
- [ ] Ключевые блоки страницы имеют повторяемую компонентную структуру, совместимую с другими страницами этого типа.
- [ ] В интро, коротких ответах, FAQ и CTA используются конкретные сущности, сценарии и отношения, а не размытые общие формулировки.
- [ ] Страница проверена на мобильном экране.
- [ ] Нет пустых блоков, `lorem ipsum`, служебных заглушек, эмодзи-спама и декоративного мусора.

## 28. Что Мониторить После Публикации

- [ ] Индексация в Search Console.
- [ ] Поисковые запросы, показы, клики, CTR и средняя позиция.
- [ ] Внешний вид в поиске по rich results, если они применимы.
- [ ] Страницы с высокими показами и низким CTR.
- [ ] Страницы с ростом показов, но плохой позицией.
- [ ] Страницы с низким вовлечением или слабым переходом на коммерческие страницы.
- [ ] Новые каннибализации внутри `articles`, `series`, `products`.
