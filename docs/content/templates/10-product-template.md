# Шаблон Страницы Товара

Обновлено: 2026-06-19.

Использовать для новых карточек товаров в `content/products/<series>/<model>/index.md` и `index.ru.md`.

Как пользоваться новичку: каждая модель/цвет/материал живет в своей папке как отдельная page bundle. Внутри должны быть украинский `index.md`, русский `index.ru.md` и точные изображения этого варианта. Не копируй характеристики или primary image из другой модели, если они не подтверждены для этой версии.

Целевой стандарт для проекта `Aerocool`: одна товарная страница обычно пишется в формате `6000+` знаков основного текста на каждую языковую версию. Объем набирается через сценарии использования, характеристики, сравнения, FAQ и коммерческий следующий шаг, а не через повторение рекламных формулировок.

Для `index.md` использовать локальные URL вида `/products/...`, `/contact/`. Для `index.ru.md` использовать `/ru/products/...`, `/ru/contact/`. Если вторичное inline-изображение позже вставляется через `seo-image` и в перевод, локализовать `alt` и `title`; параметр `jsonld` не добавлять.

Текущий рабочий паттерн для product pages:

- `image` — для SEO, OG, Twitter и schema;
- `cover.image` — для preview в листингах;
- `products/gallery.html` — для основного видимого изображения и дополнительных кадров page bundle.

Если `image` или fallback `cover.image` указывает на отсутствующий файл либо на ресурс, который Hugo не может обработать как processable image, product gallery должна остановить сборку. Это правильное поведение: лучше получить build error, чем production-страницу с битой LCP-картинкой.

Первое товарное изображение не вставлять через shortcode `seo-image`. Товарный шаблон сам берет `image` из front matter, выводит первый кадр через [layouts/_partials/products/gallery.html](../../../layouts/_partials/products/gallery.html), ставит `loading="eager"` и `fetchpriority="high"`, а [layouts/_partials/_seo/lcp-image-preload.html](../../../layouts/_partials/_seo/lcp-image-preload.html) выводит matching responsive preload в `<head>` с gallery `sizes`.

Если в описании товара позже нужна вторичная inline-иллюстрация, ее можно добавить через `seo-image`, но только как lazy image: `loading="lazy"`, `preload=false`, `fetchpriority=auto`.

## Product Image Gate

Primary image обязано показывать конкретный вариант из front matter: ту же серию, модель, расцветку и материал. Даже технически корректный файл **2000x2000** нельзя использовать для нескольких SKU, если в реальности они выглядят по-разному.

Минимальный целевой набор gallery:

1. `01-front.*` — точный front three-quarter кадр на белом или светлом нейтральном фоне;
2. `02-side.*` — боковой профиль;
3. `03-back.*` — спинка и задняя конструкция;
4. `04-material.*` — реальный macro материала;
5. `05-controls.*` или `05-mechanism.*` — рычаги, подлокотники, поясничная поддержка или механизм;
6. `06-in-scale.*` — кресло возле стола/монитора для оценки масштаба;
7. `07-dimensions.*` — необязательная точная схема размеров, только по подтвержденным данным.

Запрещены `TEST`, placeholder, mockup label, watermark поставщика, candidate-файлы, случайный AI-текст и один побайтно одинаковый primary-файл для разных вариантов. Полный контракт находится в [ДНК текстов и изображений страниц](../79-page-content-design-dna-2026.md), а текущее состояние — в [полном аудите content/image слоя](../../audits/80-2026-06-19-full-site-content-image-audit.md).

## Редакционные Правила

- Видимый `H1` рендерит шаблонный слой; в теле товарной страницы не добавлять markdown `# H1`.
- Если SEO `title` получается длинным, используйте более короткий `h1`, например `Aerocool <MODEL>`.
- В видимом тексте товарной страницы не использовать обратные кавычки для inline-code. Точные технические обозначения, характеристики, SKU/MPN/GTIN, размеры, рейтинги и значения из таблиц выделять обычным жирным форматом: `**11D**`, `**SYNC5**`, `**75 мм**`. Широкие коммерческие SEO-фразы писать обычным текстом или обычным жирным выделением.
- Не выдумывать характеристики, которые не подтверждены производителем.
- Не путать `Racer`, `Loft Air`, `Mesh`.
- Не писать общий рекламный текст без конкретики.
- В `description` и `summary` упоминать реальные отличительные особенности.
- В `title`, `description`, `summary` и `tags` учитывать релевантный коммерческий кластер: `игровое кресло`, `офисное кресло`, `компьютерное кресло`, если это соответствует конкретной модели.
- Внутри текста раскрывать не только “что есть”, но и “кому это подходит”.
- Front matter товарной страницы — единый источник правды для merchant facts в `Product` JSON-LD: цена, SKU, MPN, GTIN, гарантия, наличие, доставка, возврат и способы оплаты.
- Для рейтинга и отзывов целевой источник правды — approved отзывы из `Netlify Database`, выгруженные в `data/generated/reviews.json` на этапе build.
- Коммерческие данные, которые используются в `Product` JSON-LD, должны быть видны пользователю через товарный шаблон: цена, SKU, MPN, GTIN, гарантия, наличие, доставка, возврат и оплата выводятся из front matter. Не дублируйте эти значения ручным списком в markdown-теле, чтобы не создавать расхождение между content, HTML и schema.
- Рейтинг, количество отзывов и `Review`-разметку показывать только после появления approved отзывов, публично видимых на странице.
- Если меняются доставка, возврат, способы оплаты, цена, наличие или гарантия, сначала обновлять product front matter, затем синхронизировать видимый коммерческий блок и `/faq/`, если изменение относится ко всему policy-layer.
- Для квадратных фронтальных изображений товара нормален размер `2000x2000`; если используется широкий hero-кадр, под него можно задать другую целевую пропорцию.
- Размер сам по себе не подтверждает качество: product primary должен быть уникальным для фактического варианта, а gallery — не содержать test/placeholder assets.
