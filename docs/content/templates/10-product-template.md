# Шаблон Страницы Товара

Обновлено: 2026-06-24.

Использовать для новых карточек товаров в `content/products/<series>/<model>/index.md` и `index.ru.md`.

Как пользоваться новичку: каждая модель, цвет и материал живут в своей папке как отдельная папка страницы Hugo (`page bundle`). Внутри должны быть украинский `index.md`, русский `index.ru.md` и точные изображения этого варианта. Не копируй характеристики или главное изображение из другой модели, если они не подтверждены для этой версии.

Целевой стандарт для проекта `Aerocool`: одна товарная страница обычно пишется в формате `6000+` знаков основного текста на каждую языковую версию. Объем набирается через сценарии использования, характеристики, сравнения, FAQ и коммерческий следующий шаг, а не через повторение рекламных формулировок.

Для `index.md` использовать локальные URL вида `/products/...`, `/contact/`. Для `index.ru.md` использовать `/ru/products/...`, `/ru/contact/`. Если вторичное inline-изображение позже вставляется через `seo-image` и в перевод, локализовать `alt` и `title`; параметр `jsonld` не добавлять.

Текущий рабочий шаблон для товарных страниц:

- `image` — для SEO, OG, Twitter и schema;
- `cover.image` — для preview в листингах;
- `products/gallery.html` — для основного видимого изображения и дополнительных кадров папки страницы.

Если `image` или резервное поле `cover.image` указывает на отсутствующий файл либо на ресурс, который Hugo не может обработать как изображение, товарная галерея должна остановить сборку. Это правильное поведение: лучше получить ошибку сборки, чем production-страницу с неработающим LCP-изображением.

Первое товарное изображение не вставлять через shortcode `seo-image`. Товарный шаблон сам берет `image` из front matter, выводит первый кадр через [layouts/_partials/products/gallery.html](../../../layouts/_partials/products/gallery.html), ставит `loading="eager"` и `fetchpriority="high"`, а [layouts/_partials/_seo/lcp-image-preload.html](../../../layouts/_partials/_seo/lcp-image-preload.html) выводит соответствующий адаптивный preload в `<head>` с `sizes` галереи.

Если в описании товара позже нужна вторичная встроенная иллюстрация, ее можно добавить через `seo-image`, но только с отложенной загрузкой: `loading="lazy"`, `preload=false`, `fetchpriority=auto`.

## Контроль Товарного Изображения

Главное изображение обязано показывать конкретный вариант из front matter: ту же серию, модель, расцветку и материал. Даже технически корректный файл **2000x2000** нельзя использовать для нескольких SKU, если в реальности они выглядят по-разному.

### Имена Файлов Product Gallery

Для новых и заменяемых товарных изображений использовать одинаковые имена файлов во всех папках товаров. Папка уже содержит серию, модель и цвет, поэтому не нужно дублировать модель, SKU или цвет в имени каждого файла.

Целевой порядок файлов:

1. `01-front.webp` — главный кадр для карточки товара, SEO/OG/schema, LCP и первого кадра галереи;
2. `02-front-3q.webp` — фронтальный ракурс три четверти;
3. `03-side.webp` — боковой профиль;
4. `04-back-3q.webp` — задний ракурс три четверти;
5. `05-back.webp` — спинка и задняя конструкция;
6. `06-material.webp` — реальный close-up материала;
7. `07-controls.webp` — рычаги, подлокотники, поясничная поддержка или видимые элементы управления;
8. `08-mechanism.webp` — механизм наклона, основание или технический узел, если он отличается от `07-controls.webp`;
9. `09-in-scale.webp` — кресло возле стола, монитора или другого понятного объекта масштаба;
10. `10-dimensions.webp` или `10-dimensions.svg` — необязательная точная схема размеров только по подтвержденным данным.

Правила именования:

- использовать только латиницу, lowercase и kebab-case;
- использовать двухзначный номер `01`, `02`, `03`, чтобы порядок галереи был стабильным;
- не использовать пробелы, кириллицу, большие буквы, `final`, `new`, `v2`, `test`, `candidate`;
- не оставлять рядом дубликаты одной картинки в разных форматах, например `01-front.png` и `01-front.webp`, если это один и тот же кадр;
- для временно сохраненного официального исходника `.png` или `.jpg` можно оставить тот же stem, например `01-front.png`, но финальный целевой стандарт новых product assets — WebP.

Минимальный целевой набор gallery по смыслу:

1. главный кадр;
2. фронтальный ракурс три четверти;
3. боковой профиль;
4. задний ракурс;
5. материал;
6. органы управления или механизм;
7. кресло в масштабе;
8. необязательная схема размеров.

Запрещены `TEST`, заглушки, тестовые подписи макета, водяные знаки поставщика, файлы-кандидаты, случайный AI-текст и один побайтно одинаковый главный файл для разных вариантов. Полный контракт находится в [ДНК текстов и изображений страниц](../79-page-content-design-dna-2026.md), а текущее состояние — в [полном аудите слоя текстов и изображений](../../audits/80-2026-06-19-full-site-content-image-audit.md).

## Редакционные Правила

- Видимый `H1` рендерит шаблонный слой; в теле товарной страницы не добавлять markdown `# H1`.
- Если SEO `title` получается длинным, используйте более короткий `h1`, например `Aerocool <MODEL>`.
- В видимом тексте товарной страницы не использовать обратные кавычки для inline-code. Точные технические обозначения, характеристики, SKU/MPN/GTIN, размеры, рейтинги и значения из таблиц выделять обычным жирным форматом: `**11D**`, `**SYNC5**`, `**75 мм**`. Широкие коммерческие SEO-фразы писать обычным текстом или обычным жирным выделением.
- Не выдумывать характеристики, которые не подтверждены производителем.
- Не путать `Racer`, `Loft Air`, `Mesh`.
- Не писать общий рекламный текст без конкретики.
- В `description` и `summary` упоминать реальные отличительные особенности.
- В `title`, `description`, `summary` и `tags` учитывать релевантный коммерческий кластер: `игровое кресло`, `офисное кресло`, `компьютерное кресло`, если это соответствует конкретной модели.
- Внутри текста раскрывать не только “что есть”, но и “кому это подходит”.
- Front matter товарной страницы — единый источник правды для коммерческих фактов в `Product` JSON-LD: цена, SKU, MPN, GTIN, гарантия, наличие, доставка, возврат и способы оплаты.
- Для рейтинга и отзывов целевой источник правды — одобренные отзывы из `Netlify Database`, выгруженные в `data/generated/reviews.json` на этапе сборки.
- Коммерческие данные, которые используются в `Product` JSON-LD, должны быть видны пользователю через товарный шаблон: цена, SKU, MPN, GTIN, гарантия, наличие, доставка, возврат и оплата выводятся из front matter. Не дублируйте эти значения ручным списком в теле Markdown, чтобы не создавать расхождение между контентом, HTML и schema.org.
- Рейтинг, количество отзывов и `Review`-разметку показывать только после появления approved отзывов, публично видимых на странице.
- Если меняются доставка, возврат, способы оплаты, цена, наличие или гарантия, сначала обновлять front matter товара, затем синхронизировать видимый коммерческий блок и `/faq/`, если изменение относится ко всей политике сайта.
- Для квадратных фронтальных изображений товара нормален размер `2000x2000`; если используется широкий hero-кадр, под него можно задать другую целевую пропорцию.
- Размер сам по себе не подтверждает качество: главное изображение товара должно быть уникальным для фактического варианта, а галерея не должна содержать тестовые файлы или заглушки.
