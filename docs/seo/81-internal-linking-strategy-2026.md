## 2. Официальные Принципы Google На 2026 Год

Текущая система проекта сверена с официальными документами Google Search Central, доступными и актуальными на дату аудита 2026-06-21.

Основные принципы:

1. Внутренняя ссылка должна быть обычным HTML-элементом `<a>` с реальным `href`.
2. Анкор должен быть кратким, описательным и понятным без необходимости угадывать страницу назначения.
3. Важная страница должна иметь хотя бы одну ссылку с другой страницы сайта.
4. Универсального правильного количества ссылок на странице не существует.
5. Категории должны вести ко всем товарам, которые нужно обнаружить и индексировать.
6. Относительная важность страницы зависит в том числе от количества входящих ссылок и глубины перехода.
7. Пагинация должна использовать отдельные URL и последовательные ссылки, доступные для сканирования.
8. Внешние ссылки допустимы и полезны, когда дают пользователю источник или дополнительный контекст.
9. `nofollow` не нужно автоматически добавлять всем внешним ссылкам; для рекламы используется `sponsored`, для пользовательских ссылок — `ugc` или `nofollow`.
10. `noindex` исключает страницу из результатов Google после повторного обхода независимо от качества внутренних ссылок.

Официальные источники находятся в разделе 17.

Аудит выполнен по свежей локальной сборке:


Проверены:

- контекстные ссылки внутри основного содержимого;
- блоки `Что посмотреть дальше`;
- пагинация и `/page/1/` aliases;
- canonical, sitemap и взаимные `hreflang`;
- переключатели вариантов;
- ссылки страницы на саму себя из текущего состояния интерфейса;
- ссылки на локальные `#fragment`, `mailto:` и `tel:`;
- внешние ссылки;
- действующие `robots` и canonical главной страницы.

Ограничения аудита:

- данные Google Search Console и журналы сканирования не подключены;
- автоматический link-audit скрипт в проекте не используется: контроль ссылок выполняется вручную после production-сборки;
- метрики кликов пользователей и вспомогательных конверсий отсутствуют;
- относительный SEO-вес ссылки нельзя измерить одной локальной метрикой.

Выпадающий каталог для компьютеров и группа каталога в мобильном меню используют обычные `<a href>`. JavaScript управляет только поведением интерфейса и не является единственным источником URL.

### Подвал Сайта

[footer.html](../../layouts/_partials/footer.html) содержит основные разделы, серии и пять внешних соцпрофилей.

Глобальный слой уже составляет около **59%** всех вхождений внутренних ссылок. Поэтому подвал и шапку сайта нельзя расширять для каждого нового SEO-приоритета. Новые важные связи добавляются прежде всего в основное содержимое или тематические блоки связанных материалов.

### Хлебные Крошки

[breadcrumbs.html](../../layouts/_partials/breadcrumbs.html) строит видимую иерархию:

Название каждого узла приходит из [breadcrumb-label.html](../../layouts/_partials/breadcrumb-label.html), поэтому видимые хлебные крошки и schema.org `BreadcrumbList` используют единое правило названий.

## 9. Контракт Анкоров

Каждый анкор должен:

- описывать страницу назначения;
- оставаться естественным для текущего предложения;
- быть достаточно коротким;
- не повторять механически один коммерческий ключ;
- быть понятным при чтении без соседнего текста;
- совпадать с языком текущей страницы.

Допустимые примеры:

- `порівняння SKY Light і SKY 360`;
- `гид по механизмам SYNC4 и SYNC5`;
- `Aerocool WING Racer Black`;
- `контакти Aerocool Україна`.

Нежелательные примеры без уточнения назначения:

- `здесь`;
- `читать`;
- `подробнее`;
- `перейти`;
- `статья`.

Товарная карточка визуально сохраняет короткий призыв `Подробнее` / `Детальніше`, но его доступное имя включает `$page.Title` через `sr-only`. После исправления неинформативных анкоров без названия модели: **0**.

`sr-only` используется только для честного доступного имени элемента. Запрещено помещать туда списки ключевых слов или текст, которого пользователь не ожидает от ссылки.

### Разнообразие Анкоров

Между индексируемыми canonical URL обнаружено **1 630** вхождений ссылок в основном содержимом и **366** уникальных доступных имен. Самые частые значения — `WING`, `XTAL` и `SKY`: это естественные названия сущностей и серий, а не искусственные коммерческие анкоры с точным вхождением.

Анкорное разнообразие оценивается по смыслу, а не по требованию каждый раз писать новую формулировку. Название товара или серии можно повторять, если оно точно описывает назначение ссылки.

## 10. UK/RU-Паритет

Текущий результат: **0** расхождений графа и **0** ошибок reciprocal `hreflang` на **96** индексируемых canonical URL.

Обязательные правила:

1. UK-страница ссылается на UK URL без `/ru/`.
2. RU-страница ссылается на соответствующий `/ru/` URL.
3. Front matter `related_*` хранит языконезависимые пути контента; шаблон подставляет язык.
4. При ручной Markdown-ссылке обе языковые версии редактируются одновременно.
5. Переключатели вариантов и хлебные крошки строятся из страниц текущего языка.
6. После изменения URL нужно проверить canonical, hreflang, sitemap и обе версии всех входящих ссылок.

`hreflang` выводит пары `uk-UA` и `ru-UA` для локализованных страниц. `x-default` сейчас не используется и не является обязательным для текущей архитектуры с украинской версией по умолчанию.

## 11. Пагинация

Текущая реализация:

- страницы `/page/2/`, `/articles/page/2/`, `/news/page/2/` имеют собственные URL;
- переходы между страницами выводятся обычными `<a href>`;
- страницы пагинации имеют canonical на самих себя;
- страницы 2+ получают `noindex,follow`;
- alias URL `/page/1/` получают canonical на первую страницу и `noindex,nofollow`;
- фильтры и сортировка не меняют URL.

В production-сборке присутствуют **14** alias URL вида `/page/1/` и **14** страниц пагинации 2+. Alias URL корректно закрыты и канонизированы на первый листинг. Все страницы 2+ имеют canonical на самих себя, но остаются `noindex,follow`.

Google рекомендует отдельный URL и canonical на саму себя для каждой страницы пагинации. Текущий `noindex,follow` является сознательной проектной политикой, но его нужно пересмотреть при росте каталога или архива: для длинных последовательностей индексируемая пагинация с canonical на саму себя может быть более устойчивым способом обнаружения страниц.

Сейчас риска потери товаров или материалов нет: все **96** основных URL находятся в sitemap, а целевые страницы доступны через категории и последовательные ссылки.

Footer использует:

```html
target="_blank" rel="noopener noreferrer me"
```

Это корректно для безопасности новой вкладки и связи с официальным профилем.

В редакционных статьях сейчас нет обычных внешних ссылок на источники. Массово добавлять их не нужно. Ссылки на внешние первичные источники полезны там, где текст опирается на технический стандарт, исследование, медицинское ограничение или официальную документацию производителя.

Правила:

- не добавлять внешнюю ссылку только ради SEO;
- объяснять пользователю, куда ведет ссылка;
- не использовать `nofollow` для доверенного источника без причины;
- использовать `sponsored` для оплаченной ссылки;
- использовать `ugc` или `nofollow` для пользовательских ссылок;
- регулярно проверять переадресации, ответы 404 и смену владельца домена.

URL schema.org, пространства имен XML, `@id` и `sameAs` внутри JSON-LD не являются обычными видимыми ссылками и проверяются отдельным процессом schema.org и сущностей.

## 13. Что Нельзя Делать

- Добавлять каждую приоритетную страницу в подвал.
- Создавать скрытые списки коммерческих анкоров.
- Использовать навигацию только через JavaScript без реального `href`.
- Ссылаться на `/page/1/`, alias URL или URL переадресации.
- Вести украинскую страницу на русский URL и наоборот без переключателя языка.
- Добавлять десятки нерелевантных ссылок в блок связанных материалов.
- Создавать индексируемые URL фильтров и сортировки без отдельной SEO-стратегии.
- Добавлять `product_group_id` одиночному товару ради ссылок между вариантами.
- Использовать одинаковый анкор с точным вхождением во всех входящих ссылках.
- Считать sitemap заменой внутренней навигации.
- Считать количество ссылок самостоятельной целью.

## 14. Приоритеты После Аудита

### P0. Индексируемость Production-Сборки

Настроить отдельный Netlify production context с `hugo --environment production`, сохранив `development` и `noindex,nofollow` для `dev` Branch Deploy.

После production-развертывания проверить:

- главную;
- каталог;
- серию;
- товар;
- статью;
- новость;
- UK/RU пару;
- индекс sitemap.

### P1. Ручной Аудит Ссылочного Графа

Статус: активный постоянный процесс.

После production-сборки проверять вручную:

- неработающие ссылки;
- страницы без входящих ссылок;
- ссылки на alias URL и переадресации;
- расхождения canonical и sitemap;
- пустые и неинформативные анкоры;
- глубину более трех кликов;
- UK/RU-паритет;
- прямую доступность всех товаров из каталога и серий;
- ссылки внутри групп вариантов;
- взаимные `hreflang`;
- неработающие `#fragment`, некорректные `mailto:` и `tel:`;
- лишние ссылки страницы на саму себя из текущего состояния;
- ошибки HTTP у внешних ссылок.

Текущий baseline ручной проверки зафиксирован в [аудите `84`](../audits/84-2026-06-24-full-link-content-seo-audit.md): 0 битых внутренних ссылок, 0 сломанных фрагментов, 0 orphan URL, 0 sitemap/canonical расхождений и 0 `hreflang` ошибок.

### P2. Политика Пагинации

До существенного роста каталогов и архивов принять отдельное решение по **14** страницам пагинации 2+. Наиболее близкий к рекомендации Google вариант — `index,follow` с canonical на саму себя для последовательных URL листинга. Если сохраняется `noindex,follow`, это должно оставаться осознанной политикой с контролем обнаружения через Search Console и журналы сканирования.

### P2. Редакционные Внешние Источники

Для новых технических утверждений добавлять ссылку на официальную спецификацию товара, стандарт или первичное исследование, если без нее утверждение нельзя проверить. Не создавать массовый блок внешних ссылок на всех страницах.

### P2. Приоритетные Товары На Главной

Если бизнес-команда подтвердит лидеров продаж или стратегические SKU, можно вывести 2-4 прямые ссылки с главной. Не выбирать товары автоматически по неподтвержденному рейтингу или временной позиции в списке.

### P3. Проверка В Search Console

После включения индексации использовать реальные данные:

- усиливать URL с показами и позициями примерно 8-20;
- смотреть расхождения между запросом и целевой страницей;
- оценивать изменение показов, кликов и средней позиции;
- не менять одновременно все кластеры без возможности сравнить результат.

### P3. Очистка Ссылок Страницы На Саму Себя

При следующей плановой правке шапки сайта, переключателя языка, пагинации или переключателей вариантов заменить ссылки текущего состояния на неинтерактивные элементы там, где переход приводит на тот же URL. Это улучшение семантики и UX, а не срочное SEO-исправление.

## 15. Чеклист После Изменения Ссылок

- [ ] Обе языковые версии изменены синхронно.
- [ ] Целевой URL существует в production-сборке.
- [ ] Ссылка ведет сразу на canonical URL без переадресации.
- [ ] Анкор понятен без соседнего текста.
- [ ] Ссылка имеет реальный `<a href>`.
- [ ] Локальный `#fragment` существует в сгенерированном HTML.
- [ ] Контекст объясняет, зачем переходить.
- [ ] Новая ссылка не дублирует бессмысленно соседнюю ссылку.
- [ ] Ссылка текущего состояния не ведет без необходимости на тот же URL.
- [ ] Поле `related_*` подтверждено видимым содержанием.
- [ ] Новый товар доступен из каталога и серии.
- [ ] Новый вариант связан с группой только при реальном ProductGroup.
- [ ] Не появился новый URL фильтра или сортировки.
- [ ] Пагинация сохраняет последовательные ссылки.
- [ ] Sitemap и canonical согласованы.
- [ ] UK/RU `hreflang` взаимны и ведут на canonical URL.
- [ ] `noindex` остается только там, где это предусмотрено политикой.
- [ ] Внешняя ссылка проверена и имеет подходящий `rel`.

Минимальные действующие проверки:

```bash
./scripts/script_check.sh
npm run build:production
```

После production-сборки вручную проверять итоговый сайт: ключевые URL, sitemap, canonical, `hreflang`, локальные якоря, карточки, breadcrumbs, related-блоки и целевые страницы.
