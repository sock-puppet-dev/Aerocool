## 2. Официальные Принципы Google На 2026 Год

Текущая система проекта сверена с официальными документами Google Search Central, доступными и актуальными на дату аудита 2026-06-21.

Основные принципы:

1. Внутренняя ссылка должна быть обычным HTML-элементом `<a>` с реальным `href`.
2. Анкор должен быть кратким, описательным и понятным без необходимости угадывать страницу назначения.
3. Важная страница должна иметь хотя бы одну ссылку с другой страницы сайта.
4. Универсального правильного количества ссылок на странице не существует.
5. Категории должны вести ко всем товарам, которые нужно обнаружить и индексировать.
6. Относительная важность страницы зависит в том числе от количества входящих ссылок и глубины перехода.
7. Пагинация должна использовать отдельные URL и последовательные crawlable-ссылки.
8. Внешние ссылки допустимы и полезны, когда дают пользователю источник или дополнительный контекст.
9. `nofollow` не нужно автоматически добавлять всем внешним ссылкам; для рекламы используется `sponsored`, для пользовательских ссылок — `ugc` или `nofollow`.
10. `noindex` исключает страницу из результатов Google после повторного обхода независимо от качества внутренних ссылок.

Официальные источники находятся в разделе 17.

## 3. Охват И Методика Аудита

Аудит выполнен по свежей локальной сборке:

```bash

```

Проверены:

- UK и RU версии;

- карточки листингов и товарные карточки;
- contextual links внутри body;
- блоки `Что посмотреть дальше`;
- пагинация и `/page/1/` aliases;
- canonical, sitemap и взаимные `hreflang`;
- variant swatches;
- self-links текущего состояния интерфейса;
- ссылки на локальные `#fragment`, `mailto:` и `tel:`;
- внешние anchor links;
- live `robots` и canonical главной страницы.

Ограничения аудита:

- данные Google Search Console и crawl logs не подключены;
- в репозитории пока нет постоянного `links:audit` скрипта;
- метрики кликов пользователей и assisted conversions отсутствуют;
- относительный SEO-вес ссылки нельзя измерить одной локальной метрикой.

## 6. Реализованная Архитектура Ссылок

### Header И Mobile Menu

- каталог;

- WING;
- XTAL;
- About;
- Articles;

Desktop catalog flyout и mobile catalog group используют обычные `<a href>`. JavaScript управляет только поведением интерфейса и не является единственным источником URL.

### Footer

[footer.html](../../layouts/_partials/footer.html) содержит основные разделы, серии и пять внешних соцпрофилей.

Sitewide-слой уже составляет около **59%** всех внутренних anchor occurrences. Поэтому footer и header нельзя расширять для каждого нового SEO-приоритета. Новые важные связи добавляются прежде всего в body или тематические related-блоки.

### Breadcrumbs

[breadcrumbs.html](../../layouts/_partials/breadcrumbs.html) строит видимую иерархию:

```text
Главная -> раздел -> серия -> товар
```

Название каждого узла приходит из [breadcrumb-label.html](../../layouts/_partials/breadcrumb-label.html), поэтому visible breadcrumbs и schema.org `BreadcrumbList` используют один label contract.

### Каталог И Серии

[products/list.html](../../layouts/products/list.html) и [products/card.html](../../layouts/_partials/products/card.html) обеспечивают прямые ссылки:

- `/products/` -> все **12** товаров текущего языка;
- `/products/sky/` -> **2** модели;
- `/products/wing/` -> **5** моделей;
- `/products/xtal/` -> **5** моделей.

Фильтры и сортировка static-first и не создают индексируемые filter URLs. Это предотвращает дубли и crawl traps.

### Варианты Товаров

[variant-swatches.html](../../layouts/_partials/products/variant-swatches.html) связывает реальные варианты одной модели через `product_group_id`.

Проверено **16** страниц variant-групп двух языков. Каждый участник группы имеет ссылку на соседний вариант. Одиночные товары не получают искусственный `product_group_id`.

### Articles И News

Хабы `/articles/` и `/news/` используют crawlable карточки и пагинацию. В body материалов присутствуют ссылки на:

- серии;
- товары;
- сравнения;
- FAQ;
- Contact;
- соседние гайды.

### Related-Блок

[post-footer-navigation.html](../../layouts/_partials/post-footer-navigation.html) строит блок `Что посмотреть дальше` для products, articles и news.

Источники сигналов:

- `related_series`;
- `related_products`;
- `related_articles`;
- `related_news`;
- category/categories/tags;
- `product_group_id`;
- `about_entities`;
- резервное определение серии по структурированному тексту страницы.

Явные `related_*` ставятся первыми, но не заменяют видимую контекстную причину связи в тексте страницы.

## 9. Контракт Анкоров

Каждый анкор должен:

- описывать страницу назначения;
- оставаться естественным для текущего предложения;
- быть достаточно коротким;
- не повторять механически один коммерческий ключ;
- быть понятным при чтении без соседнего текста;
- совпадать с языком текущей страницы.

Допустимые примеры:

- `порівняння SKY Lite і SKY 360`;
- `гид по механизмам SYNC4 и SYNC5`;
- `Aerocool WING Racer Black`;
- `контакти Aerocool Україна`.

Нежелательные примеры без уточнения назначения:

- `здесь`;
- `читать`;
- `подробнее`;
- `перейти`;
- `статья`.

Товарная карточка визуально сохраняет короткий CTA `Подробнее` / `Детальніше`, но его accessible name включает `$page.Title` через `sr-only`. После исправления generic anchors без названия модели: **0**.

`sr-only` используется только для честного доступного имени элемента. Запрещено помещать туда списки ключевых слов или текст, которого пользователь не ожидает от ссылки.

### Разнообразие Анкоров

Между индексируемыми canonical URL обнаружено **1 630** body-link occurrences и **366** уникальных доступных имен. Самые частые значения — `WING`, `XTAL` и `SKY`: это естественные названия сущностей и серий, а не искусственные exact-match коммерческие анкоры.

Анкорное разнообразие оценивается по смыслу, а не по требованию каждый раз писать новую формулировку. Название товара или серии можно повторять, если оно точно описывает назначение ссылки.

### Self-Links

Обнаружено **188** ссылок страницы на саму себя:

- **172** находятся в global/UI-слое: текущий язык в переключателе, активные пункты навигации и текущий номер пагинации;
- **16** находятся в body и соответствуют текущему variant swatch на страницах реальных ProductGroup.

Это не создает crawl trap и не считается SEO-ошибкой. Для более чистой семантики текущий язык, текущий номер страницы и активный swatch в будущем можно выводить не ссылкой, а `span` или другим неинтерактивным элементом с сохранением доступного current-state обозначения.

### Fragment, Email И Телефон

Проверено **1 488** ссылок на локальные `#fragment`: все ведут на существующий `id` целевой страницы. Пустых `href`, ссылок-заглушек `href="#"` и `javascript:` нет.

Контактный слой содержит **6** `mailto:` и **2** `tel:` occurrences в двух языковых версиях. Видимый адрес или номер совпадает с назначением ссылки.

## 10. UK/RU-Паритет

Текущий результат: **0** расхождений графа и **0** ошибок reciprocal `hreflang` на **96** индексируемых canonical URL.

Обязательные правила:

1. UK-страница ссылается на UK URL без `/ru/`.
2. RU-страница ссылается на соответствующий `/ru/` URL.
3. Front matter `related_*` хранит языконезависимые content paths; шаблон подставляет язык.
4. При ручной markdown-ссылке обе языковые версии редактируются одновременно.
5. Variant swatches и breadcrumbs строятся из страниц текущего языка.
6. После изменения URL нужно проверить canonical, hreflang, sitemap и обе версии всех входящих ссылок.

`hreflang` выводит пары `uk-UA` и `ru-UA` для локализованных страниц. `x-default` сейчас не используется и не является обязательным для текущей архитектуры с украинской default-версией.

## 11. Пагинация

Текущая реализация:

- страницы `/page/2/`, `/articles/page/2/`, `/news/page/2/` имеют собственные URL;
- переходы между страницами выводятся обычными `<a href>`;
- страницы пагинации имеют self canonical;
- страницы 2+ получают `noindex,follow`;
- `/page/1/` aliases получают canonical на первую страницу и `noindex,nofollow`;
- фильтры и сортировка не меняют URL.

В production-сборке присутствуют **14** alias URL вида `/page/1/` и **14** страниц пагинации 2+. Alias URL корректно закрыты и канонизированы на первый листинг. Все страницы 2+ имеют self canonical, но остаются `noindex,follow`.

Google рекомендует отдельный URL и self canonical для каждой страницы пагинации. Текущий `noindex,follow` является сознательной проектной политикой, но его нужно пересмотреть при росте каталога или архива: для длинных последовательностей индексируемая self-canonical pagination может быть более устойчивым способом discovery.

Сейчас риска потери товаров или материалов нет: все **96** основных URL находятся в sitemap, а целевые страницы доступны через категории и последовательные ссылки.

В rendered HTML обнаружено:

- все URL являются global social profiles;
- каждый URL присутствует на **116** страницах;
- все пять URL вернули HTTP **200** на 2026-06-21.

Профили:

- Facebook;
- Instagram;
- TikTok;
- YouTube;
- X.

Footer использует:

```html
target="_blank" rel="noopener noreferrer me"
```

Это корректно для безопасности новой вкладки и связи с официальным профилем.

В редакционных статьях сейчас нет обычных внешних anchor citations. Массово добавлять их не нужно. Ссылки на внешние первичные источники полезны там, где текст опирается на технический стандарт, исследование, медицинское ограничение или официальную документацию производителя.

Правила:

- не добавлять внешнюю ссылку только ради SEO;
- объяснять пользователю, куда ведет ссылка;
- не использовать `nofollow` для доверенного источника без причины;
- использовать `sponsored` для оплаченной ссылки;
- использовать `ugc` или `nofollow` для пользовательских ссылок;
- регулярно проверять redirects, 404 и смену владельца домена.

URL schema.org, XML namespaces, `@id` и `sameAs` внутри JSON-LD не являются обычными видимыми anchor links и проверяются отдельным schema/entity workflow.

## 13. Что Нельзя Делать

- Добавлять каждую приоритетную страницу в footer.
- Создавать скрытые списки коммерческих анкоров.
- Использовать JavaScript-only navigation без реального `href`.
- Ссылаться на `/page/1/`, aliases или redirect URL.
- Вести UK-страницу на RU URL и наоборот без языкового switcher.
- Добавлять десятки нерелевантных links в related-блок.
- Создавать индексируемые filter/sort URL без отдельной SEO-стратегии.
- Добавлять `product_group_id` одиночному товару ради variant links.
- Использовать одинаковый exact-match анкор во всех входящих ссылках.
- Считать sitemap заменой внутренней навигации.
- Считать количество ссылок самостоятельной целью.

## 14. Приоритеты После Аудита

### P0. Production Indexability

Настроить отдельный Netlify production context с `hugo --environment production`, сохранив `development` и `noindex,nofollow` для `dev` Branch Deploy.

После production deploy проверить:

- главную;
- каталог;
- серию;
- товар;
- статью;
- новость;
- UK/RU пару;
- sitemap index.

### P1. Постоянный Link Graph Audit

Добавить в репозиторий поддерживаемый скрипт и npm-команду, которая после production-сборки проверяет:

- broken links;
- orphan pages;
- ссылки на aliases и redirects;
- canonical/sitemap mismatch;
- пустые и generic anchors;
- глубину более трех кликов;
- UK/RU-паритет;
- прямую доступность всех товаров из каталога и серий;
- variant-group links;
- reciprocal `hreflang`;
- broken `#fragment`, некорректные `mailto:` и `tel:`;
- лишние self-links текущего состояния;
- внешние HTTP failures.

Целевая будущая команда:

```bash
npm run links:audit
```

На 2026-06-21 эта команда еще не реализована и не должна указываться как действующая.

### P2. Политика Пагинации

До существенного роста каталогов и архивов принять отдельное решение по **14** страницам пагинации 2+. Наиболее близкий к рекомендации Google вариант — self-canonical `index,follow` для последовательных listing URL. Если сохраняется `noindex,follow`, это должно оставаться осознанной политикой с контролем discovery через Search Console и crawl logs.

### P2. Редакционные Внешние Источники

Для новых технических утверждений добавлять ссылку на официальный product sheet, стандарт или первичное исследование, если без него утверждение нельзя проверить. Не создавать массовый блок внешних ссылок на всех страницах.

### P2. Приоритетные Товары На Главной

Если команда бизнеса подтвердит лидеры продаж или стратегические SKU, можно вывести 2-4 прямые ссылки с главной. Не выбирать товары автоматически по неподтвержденному рейтингу или временной позиции в списке.

### P3. Search Console

После включения индексации использовать реальные данные:

- усиливать URL с показами и позициями примерно 8-20;
- смотреть query-page mismatch;
- оценивать изменение impressions, clicks и average position;
- не менять одновременно все кластеры без возможности сравнить результат.

### P3. Очистка Self-Links

При следующей плановой правке header, language switcher, pagination или variant swatches заменить ссылки текущего состояния на неинтерактивные элементы там, где переход приводит на тот же URL. Это улучшение семантики и UX, а не срочное SEO-исправление.

## 15. Checklist После Изменения Ссылок

- [ ] Обе языковые версии изменены синхронно.
- [ ] Целевой URL существует в production-сборке.
- [ ] Ссылка ведет сразу на canonical URL без redirect.
- [ ] Анкор понятен без соседнего текста.
- [ ] Ссылка имеет реальный `<a href>`.
- [ ] Локальный `#fragment` существует в rendered HTML.
- [ ] Контекст объясняет, зачем переходить.
- [ ] Новая ссылка не дублирует бессмысленно соседний link.
- [ ] Ссылка текущего состояния не ведет без необходимости на тот же URL.
- [ ] Related field подтвержден видимым содержанием.
- [ ] Новый товар доступен из каталога и серии.
- [ ] Новый variant связан с группой только при реальном ProductGroup.
- [ ] Не появился новый filter/sort URL.
- [ ] Пагинация сохраняет последовательные ссылки.
- [ ] Sitemap и canonical согласованы.
- [ ] UK/RU `hreflang` взаимны и ведут на canonical URL.
- [ ] `noindex` остается только там, где это предусмотрено политикой.
- [ ] Внешняя ссылка проверена и имеет подходящий `rel`.

Минимальные действующие проверки:

```bash
./scripts/script_check.sh
npm run build:production
```

После сборки дополнительно нужен полный crawl rendered HTML. До реализации `npm run links:audit` эта проверка остается отдельным аудитом, а не частью автоматического quality gate.
