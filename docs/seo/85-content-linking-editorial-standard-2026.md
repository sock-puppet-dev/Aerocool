## 1. Что Означает “Топ-1 Контент”

Для проекта Aerocool “топ-1 контент” означает страницу, которая:

- полностью закрывает один главный интент;
- дает проверяемую пользу, а не пересказ выдачи;
- объясняет выбор через факты, сценарии и ограничения;
- ведет пользователя к следующему логичному URL;
- имеет чистый `title`, `description`, `H1`, структуру `H2/H3`, изображения и schema.org;
- не заставляет пользователя возвращаться в Google с тем же вопросом;
- после индексации улучшается по данным Search Console.

Не означает:

- гарантию первого места;
- искусственное увеличение количества знаков;
- механическое добавление ключевых слов;
- массовую генерацию похожих страниц;
- замену реальных товарных доказательств AI-иллюстрациями.

## 2. Базовая Формула Любой SEO-Страницы

Каждая индексируемая страница должна отвечать на пять вопросов:

1. **Что это?** Сущность, модель, серия, тема или событие.
2. **Для кого?** Пользовательский сценарий и уровень готовности к покупке.
3. **Почему верить?** Факты, характеристики, источники, ограничения, изображение.
4. **Как выбрать или действовать?** Критерии, сравнение, проверка, следующий шаг.
5. **Куда идти дальше?** Серия, товар, статья, FAQ, контакты или новости.

Если страница не отвечает на один из этих вопросов, это не всегда ошибка, но причина должна быть понятна из типа страницы. Например, `search` и `contact-success` являются функциональными/noindex страницами и не обязаны иметь полный SEO-текст.

## 3. Контентные Слои По Типам URL

| Тип URL | Главный интент | Обязательный контент | Обязательные ссылки |
|---|---|---|---|
| Главная | Понять бренд и выбрать маршрут | Кто мы, что продаем, серии, преимущества, trust-layer | Каталог, SKY, WING, XTAL, статьи, FAQ, контакты |
| `/products/` | Широкий коммерческий выбор | Серии, материалы, регулировки, сценарии, фильтры | Серии, товары, гайды выбора, FAQ |
| Серия | Выбор линейки | Позиционирование, отличия моделей, кому подходит, кому не подходит | Товары серии, соседние серии, сравнения |
| Товар | Покупка конкретного варианта | SKU/MPN/GTIN, цена, наличие, гарантия, сценарии, ограничения, FAQ | Серия, варианты, сравнения, FAQ, контакты |
| Статья | Информационный выбор | Ответ, критерии, сравнение, проверки, ограничения | Серии, товары, связанные статьи, FAQ |
| Новость | Инфоповод | Что произошло, почему важно, что делать дальше | Новые модели, серия, статья, каталог |
| FAQ | Возражения | Прямой ответ, пояснение, актуальные факты | Каталог, товары, контакты, статьи |
| About | Доверие | Кто отвечает, как проверяются данные, официальный контекст | Каталог, контакты, ключевые статьи |
| Contact | Связь | Каналы связи, часы, сценарии обращения, форма | Каталог, FAQ, товары |

## 4. Литературная Обработка

Литературная обработка в проекте не означает “сделать красиво”. Она означает сделать текст яснее, честнее и полезнее.

Проверять каждый материал так:

- первый абзац отвечает на вопрос пользователя;
- один абзац раскрывает одну мысль;
- заголовок раздела обещает конкретную пользу;
- в разделе есть факт, критерий, сценарий, ограничение или действие;
- нет повторяющихся вводных формул;
- нет искусственных выводов в середине;
- нет абсолютов без доказательств;
- нет медицинских обещаний;
- нет имитации личного теста без фактического теста;
- украинская и русская версии синхронны по смыслу, но звучат естественно.

Удалять или переписывать:

- “важно отметить”, если дальше нет нового факта;
- “идеальное решение”, если нет критериев;
- “максимальный комфорт”, если нет измеримых или объясненных причин;
- повторы названия модели вместо реального сравнения;
- перечисление ключевых слов без связного ответа;
- одинаковые абзацы в разных товарах, где должен быть variant-specific текст.

## 5. Стандарт Ссылок В Контенте

Каждая индексируемая SEO-страница должна иметь:

- минимум один путь вверх по архитектуре: товар -> серия -> каталог;
- минимум один следующий шаг по интенту: статья -> серия/товар/FAQ/контакты;
- анкор, понятный без соседнего текста;
- ссылку на canonical URL, а не alias или redirect;
- локализованную ссылку текущего языка;
- `#fragment` только если такой `id` существует в итоговом HTML.

Не добавлять ссылку, если:

- она не помогает текущему вопросу;
- она вставлена только ради количества;
- анкор является набором коммерческих ключей;
- ссылка ведет с украинской страницы на русский URL без языкового переключателя;
- ссылка ведет на неиндексируемый служебный URL без функциональной причины.

После изменения ссылок запускать production-сборку и проверять ссылки вручную:

```bash
npm run build:production
```

Ручная проверка должна пройти по ключевым URL, sitemap, canonical, `hreflang`, локальным `#fragment`, карточкам, breadcrumbs, related-блокам и целевым страницам.

## 6. SERP-Сниппет

Целевые ориентиры:

- `title` обычно держать до **70** символов;
- `description` обычно держать до **165** символов;
- `title` должен обещать реальную страницу, а не преувеличение;
- `description` должен объяснять пользу и сущность страницы;
- `H1` должен совпадать с интентом, но не обязан дословно повторять `title`;
- `linkTitle` использовать для короткой навигации и breadcrumbs.

Допустимые исключения:

- брендовый или технический термин может сделать title длиннее, если сокращение ухудшит понимание;
- служебные/noindex страницы могут иметь короткие descriptions;
- Google может переписать сниппет даже при идеальном description.

## 7. Объем И Глубина

Редакционные ориентиры проекта:

| Тип страницы | Ориентир |
|---|---:|
| Evergreen-статья | **10000+** знаков |
| Новость с органической ролью | **5000+** знаков |
| Товар | **6000+** знаков |
| Серия | **6000+** знаков |
| Хаб `/products/`, `/articles/`, `/news/` | **7000+** знаков |
| `/about/` | **10000+** знаков |

Эти числа не являются фактором ранжирования сами по себе. Если страница не дотягивает до ориентира, добирать нужно:

- критериями выбора;
- сравнением;
- практической проверкой;
- ограничениями;
- FAQ;
- подтвержденными фактами;
- ссылкой на следующий шаг.

Нельзя добирать:

- повтором интро;
- общими рекламными формулами;
- повтором ключей;
- блоком, который не меняет решение пользователя.

## 8. Factual Evidence

Каждое важное утверждение должно иметь опору:

- front matter product facts;
- официальная спецификация;
- видимый текст карточки;
- изображение конкретного товара;
- локальный бизнес-факт;
- внешний первичный источник, если речь о стандарте или техническом утверждении.

Для товаров изображение является частью доказательства. Нельзя считать карточку полностью готовой, если изображение не соответствует конкретному SKU, цвету или материалу.

## 9. Внешние Источники

Внешние ссылки добавлять не массово, а по необходимости:

- технический стандарт;
- официальная спецификация;
- медицинское или эргономическое ограничение;
- первичный источник, без которого утверждение нельзя проверить.

Не ставить `nofollow` на доверенный источник без причины. Для рекламы использовать `sponsored`, для пользовательских ссылок — `ugc` или `nofollow`.

## 10. Контроль После Изменений

Минимальный QA для контентно-ссылочной задачи:

```bash
npm run build:production
./scripts/script_check.sh
```

Дополнительно, когда задача затрагивает schema.org:

```bash
npm run entity:report
```

После публикации:

- PageSpeed Insights для ключевых URL;
- Rich Results Test или validator.schema.org для важных страниц;
- Search Console после включения индексации;
- сравнение CTR и позиций до/после правок.

## 11. Когда Создавать Новую Страницу

Новая страница нужна только если:

- текущий URL не закрывает интент;
- есть отдельный устойчивый спрос;
- страница не будет каннибализировать существующую;
- есть достаточно фактов, изображений и ссылок;
- можно поддерживать UK/RU версии.

Новая страница не нужна, если:

- это перестановка слов существующего запроса;
- блок можно добавить в существующую статью;
- интент совпадает с товаром или серией;
- нет подтвержденных фактов или изображений.

## 12. Текущие Жесткие Ограничители Роста

Пока эти пункты не закрыты, говорить о “полностью идеальном” SEO-контенте нельзя:

- Netlify production остается `noindex,nofollow`;
- товарные изображения не доказывают конкретный SKU;
- нет Search Console данных;
- нет approved review layer в публичном HTML;
- нет фактических фото trust-слоя для About/Contact.
