# Руководство по стилю документации

Обновлено: 2026-06-02.

Этот документ задает единый стандарт для всей документации проекта `Aerocool Ukraine`.

Главная цель: любой новый человек должен понять проект без устных пояснений. Документация должна объяснять не только “что делать”, но и “зачем это нужно”, “где это находится”, “что нельзя ломать” и “как проверить результат”.

## 1. Базовое Правило

Вся проектная документация пишется на русском языке.

Исключения допустимы только там, где английский текст является точным названием:

- технологии: `Hugo`, `Netlify`, `Tailwind CSS`, `PageSpeed Insights`;
- SEO-термины, которые обычно используются в оригинале: `Core Web Vitals`, `Search Console`, `schema.org`, `JSON-LD`;
- имена файлов, полей, команд и параметров: `schema_types`, `cover.image`, `npm run build`;
- официальные названия метрик или HTTP-заголовков: `LCP`, `INP`, `CLS`, `Content-Security-Policy`.

Если английский термин важен для новичка, рядом нужно дать русское пояснение.

Пример:

```md
`canonical` — это основной URL страницы, который поисковик должен считать главным.
```

## 2. Структура Любого Документа

Новый или обновленный документ должен иметь предсказуемую структуру:

1. Заголовок на русском языке.
2. Дата обновления.
3. Короткое объяснение, зачем нужен документ.
4. Блок “Как пользоваться новичку”, если документ длиннее одной страницы.
5. Основные правила или сценарии.
6. Где находятся связанные файлы проекта.
7. Что проверять после изменений.
8. Чего не делать.
9. Ссылки на связанные документы.

Не каждый короткий документ обязан иметь все разделы, но порядок должен оставаться логичным: сначала смысл, потом действия, потом проверки.

## 3. Заголовки

Заголовки пишутся по-русски и должны быть понятны без контекста.

Хорошо:

```md
# Руководство по проверке страницы через PageSpeed Insights
## 3. Как запускать проверку
## 7. Что проверять после изменений
```

Плохо:

```md
Название: PageSpeed Single URL Check Guide
Раздел: Current Status
Раздел: Implementation Backlog
```

Английский термин можно оставить в заголовке, если это имя инструмента или продукта, но поясняющие слова должны быть русскими:

```md
# Руководство по Core Web Vitals 2026 для Aerocool
```

## 4. Тон И Уровень Пояснений

Писать нужно так, будто читатель впервые открыл проект.

Это значит:

- не начинать с внутренних сокращений без расшифровки;
- не писать только список команд без объяснения результата;
- не использовать “очевидно”, “просто”, “как обычно”, если действие может быть неочевидным новичку;
- рядом с правилом давать причину, если нарушение может сломать SEO, сборку, schema.org или routing;
- отделять обязательное правило от рекомендации.

Формула хорошего объяснения:

```text
Что это такое -> где находится -> кто использует -> что изменится -> как проверить.
```

## 5. Термины И Кодовые Названия

Имена файлов, front matter поля, Hugo partials, команды и URL не переводятся.

Примеры:

- `layouts/_partials/head.html`;
- `content/products/<series>/<model>/index.md`;
- `schema_types`;
- `product_group_id`;
- `npm run build`;
- `/products/sky/360/`.

Но вокруг них должен быть русский текст:

```md
Поле `schema_types` говорит шаблонам, какие JSON-LD сущности нужно вывести на странице.
```

## 6. Строгая Структура Списков

Для процедур использовать нумерованные списки.

Для правил, запретов и справочных перечислений использовать короткие маркеры.

Если список становится длиннее 10-12 пунктов, лучше разбить его на несколько разделов.

## 7. Аудиты И Исторические Снимки

Аудит-файлы в `docs/audits/` тоже должны быть понятны новичку, но они остаются снимками своего времени.

Правила:

- новые audit-файлы писать по-русски;
- сразу после заголовка указывать дату: `Дата аудита: YYYY-MM-DD.`;
- старые audit-файлы не переписывать задним числом только ради изменения исторической оценки;
- если старый audit устарел, создать новый audit-снимок или добавить свежий статус в активный reference-документ;
- английские заголовки вроде `Scope`, `Findings`, `Status`, `Current`, `Recommended` при плановом обновлении переводить на русский.

## 8. Нумерация Файлов В `docs/`

Все файлы внутри `docs/` должны иметь глобальный двухзначный префикс в имени.

Правило:

- `01-...` — главный вход в документацию;
- `02-28` — активные руководства, справочники, чек-листы и playbook-и;
- `29+` — audit-снимки и исторические оценки;
- для audit-файлов использовать формат `NN-YYYY-MM-DD-topic-audit.md`;
- не переименовывать корневые `README.md` и `AGENTS.md`, потому что это стандартные входные файлы репозитория и Codex.

Зачем это нужно: новичок должен видеть порядок чтения прямо в имени файла, а не искать его только внутри оглавления.

Если добавляется новый документ:

1. Выбрать следующий свободный номер.
2. Назвать файл через `NN-topic.md`.
3. Добавить файл в [docs/01-documentation-map.md](/Users/stadnyk/MEGA/Aerocool/docs/01-documentation-map.md).
4. Обновить ссылки в [README.md](/Users/stadnyk/MEGA/Aerocool/README.md), [AGENTS.md](/Users/stadnyk/MEGA/Aerocool/AGENTS.md) и связанных документах.
5. Проверить локальные markdown-ссылки.

## 9. Проверка После Правок Документации

После заметных правок документации проверить:

```bash
npm run build
```

Дополнительно для крупных обновлений:

- проверить, что новые документы добавлены в `docs/01-documentation-map.md`;
- проверить, что root `README.md` ведет к полной карте документации и не противоречит новым audit-снимкам;
- проверить, что ссылки на локальные файлы открываются;
- проверить, что новые правила не противоречат `AGENTS.md`;
- проверить, что кодовые имена полей совпадают с реальными шаблонами.

## 10. Актуальный Алгоритм Обновления Документации

Этот алгоритм использовать при полном обновлении документации проекта.

1. Проверить рабочее дерево:

```bash
git status --short
```

2. Получить полный список документации:

```bash
rg --files README.md AGENTS.md docs
```

3. Проверить, что у всех рабочих документов есть дата. Исключение — шаблоны в `docs/content/templates/`.

4. Проверить, что [docs/01-documentation-map.md](/Users/stadnyk/MEGA/Aerocool/docs/01-documentation-map.md) перечисляет все документы внутри `docs/`.

5. Проверить, что root [README.md](/Users/stadnyk/MEGA/Aerocool/README.md) ведет к полной карте документации, а [AGENTS.md](/Users/stadnyk/MEGA/Aerocool/AGENTS.md) перечисляет актуальные локальные гайды.

6. Сверить документацию с фактическими файлами проекта:

- `layouts/` и [03-hugo-template-helpers.md](/Users/stadnyk/MEGA/Aerocool/docs/architecture/03-hugo-template-helpers.md);
- `content/` и [05-front-matter-reference.md](/Users/stadnyk/MEGA/Aerocool/docs/content/05-front-matter-reference.md);
- `data/`, `data/entities.yaml` и entity-документы;
- `package.json`, `mise.toml`, `netlify.toml` и deploy-документы;
- quality-документы и PageSpeed workflow.

7. Для нестабильных тем свериться с официальными источниками. Это обязательно для `Google Search`, `structured data`, `Core Web Vitals`, `Hugo`, `Netlify`, `Tailwind CSS`, `Netlify Database` и review/rating правил.

8. Исправить только найденные расхождения. Не переписывать исторический audit задним числом, если достаточно добавить свежий статус в активный документ или новый audit-снимок.

9. После правок проверить локальные markdown-ссылки, служебные англоязычные заголовки и пробельные ошибки:

```bash
git diff --check
```

10. Запустить сборку:

```bash
npm run build
```

11. Обновить текущий audit-снимок, если изменились счетчики документов, список проверенных источников, открытые риски или правила поддержки.

12. В итоговом сообщении указать:

- какие документы изменены;
- какие проверки прошли;
- какие project-gates остаются открытыми.

## 11. Чего Не Делать

Не делать:

- не писать новые подробные гайды на английском языке;
- не использовать английские заголовки ради “технического стиля”;
- не прятать важные правила только в audit-файлах;
- не создавать документ без ссылки из `docs/01-documentation-map.md`;
- не добавлять SEO/schema-правило без указания, где оно реализуется в проекте;
- не смешивать историческую оценку и текущий статус без даты.

## 12. Короткая Памятка

Документация Aerocool должна быть:

- русскоязычной;
- понятной новичку;
- структурной;
- проверяемой;
- связанной с реальными файлами проекта;
- честной про открытые риски и ограничения.
