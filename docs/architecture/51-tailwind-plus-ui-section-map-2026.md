# Карта Tailwind Plus UI-Секций Для Aerocool

Обновлено: 2026-05-22.

## Зачем Нужен Этот Документ

Этот документ фиксирует рабочую карту UI-блоков для проекта `Aerocool Ukraine` на базе трех групп Tailwind Plus:

- [Marketing UI Blocks](https://tailwindcss.com/plus/ui-blocks#product-marketing)
- [Application UI Blocks](https://tailwindcss.com/plus/ui-blocks#product-application-ui)
- [Ecommerce UI Blocks](https://tailwindcss.com/plus/ui-blocks#product-ecommerce)

Важно: Tailwind Plus используется здесь как **референс структуры и UX-паттернов**, а не как источник для механического копирования. Реализация должна учитывать текущий стек проекта: `Hugo 0.161.0`, `Tailwind CSS 4`, локальные overrides в `layouts/`, контент в `content/`, schema.org partials в `layouts/_partials/_schema` и текущий visual layer в `assets/css/main.css`.

## Базовые Правила Внедрения

| Правило | Что Это Значит Для Проекта |
|---|---|
| Сначала локальные overrides | Правки вносятся в `layouts/`, `content/`, `assets/css/main.css` и `data/`, а не напрямую в `themes/PaperMod`, если это можно решить локально. |
| Tailwind Plus не копируется слепо | Берем структуру секции, UX-логику, responsive-поведение и типы элементов, но адаптируем визуальный слой под Aerocool. |
| Единая типографика важнее копирования классов | Для заголовков, lead-текста, карточек, meta, бейджей, alert-состояний, footer, форм и кнопок использовать проектные CSS-хуки `.ui-*` из `assets/css/main.css`; прямые Tailwind `text-*`, `tracking-*` и цветовые классы оставлять только там, где они не ломают общий масштаб. |
| Блок должен решать задачу страницы | Каждый UI-блок должен помогать выбрать кресло, перейти в каталог, сравнить модели, понять условия покупки или связаться с Aerocool. |
| SEO и UX должны совпадать | Видимый текст, внутренние ссылки, product facts и JSON-LD не должны расходиться. |
| Не добавлять тяжелый JS без причины | Для базового каталога, FAQ, контактов, таблиц и карточек сначала использовать статический Hugo/Tailwind. |
| Не плодить SEO-мусорные URL | Фильтры каталога должны быть UX-инструментом, а не генератором лишних индексируемых страниц. |
| Двуязычность обязательна | Если блок появляется в `uk`, должна быть синхронная `ru`-версия, если задача явно не ограничена одним языком. |
| Baymard использовать как UX-контроль | Tailwind Plus задает визуальную структуру, а Baymard помогает проверить e-commerce UX: навигацию, списки товаров, фильтры, поиск, PDP, формы и checkout-границы. |
| Проверять сборку | После реализации запускать минимум `npm run build`; для шаблонов и CSS дополнительно проверять главную, листинг и детальную страницу в обоих языках. |

## Baymard UX 2026: Уточнения Для Aerocool

Эта секция дополняет Tailwind Plus карту практическими UX-требованиями. Tailwind Plus отвечает за форму блока, Baymard-проверка отвечает за то, помогает ли блок пользователю реально найти, сравнить и выбрать кресло.

| UX-Область | Что Проверять По Baymard-Логике | Как Применить В Aerocool | Минимальный Критерий Готовности | Приоритет |
|---|---|---|---|---|
| Homepage и category navigation | Пользователь должен быстро понять ассортимент и начать browsing без лишнего поиска. | На главной явно показать `SKY`, `WING`, `XTAL`, популярные модели и путь выбора по сценарию. | За первый экран и следующий блок пользователь видит, что сайт про кресла, какие серии есть и куда нажать дальше. | P0 |
| Mobile navigation | Основные товарные категории не должны быть глубоко спрятаны. | В mobile menu серии `SKY`, `WING`, `XTAL` показывать как быстрые товарные входы; FAQ, статьи и контакты держать ниже как secondary links. | На mobile до серии можно дойти за 1-2 действия из меню. | P1 |
| Product lists | Карточка должна помогать отсеять неподходящие товары до перехода на PDP. | В карточке показывать фото, название, цену, наличие, серию, материал, регулировки, механизм и CTA. | Пользователь может сравнить 4-6 карточек без открытия каждой страницы. | P0 |
| Варианты модели | Цвета и материалы не должны раздувать список и заставлять пользователя заново сравнивать одинаковую модель. | Если одна модель отличается только цветом или материалом, показывать swatches или компактные варианты внутри карточки, сохраняя отдельные URL страниц вариантов. | В листинге видно, что у модели есть варианты, но каталог не выглядит как набор дублей. | P1 |
| Product thumbnails | Для визуально важных товаров одного основного фото часто мало. | Для приоритетных карточек предусмотреть 2-3 изображения или явный переход в галерею: общий вид, материал, регулировка/механизм. | В карточке или на PDP пользователь видит не только фронтальное фото, но и фактуру/деталь. | P1 |
| Filters | Фильтры должны соответствовать тем атрибутам, которые видны в карточках. | Если в карточке есть материал, регулировки, серия, наличие или сценарий, по ним должен быть фильтр или понятная навигация. | Нет ситуации, когда карточка показывает важный атрибут, а отфильтровать по нему нельзя. | P1 |
| Applied filters | Пользователь должен понимать, какие условия сейчас ограничивают список. | Над списком показывать выбранные фильтры, количество найденных товаров и `сбросить`. | После выбора фильтра видно, что именно применено и как это убрать. | P1 |
| Multi-select filters | Пользователь часто сравнивает несколько значений одного типа. | Разрешить выбирать несколько материалов или сценариев, если это не ломает текущий static-first подход. | Можно сравнить, например, `Mesh` и `Loft Air`, не делая два отдельных прохода по каталогу. | P2 |
| Sorting | Сортировка ускоряет просмотр при наличии цены, популярности или новых моделей. | Добавить сортировку по релевантности/серии, цене, наличию или новизне, если данные надежны. | Сортировка не противоречит product facts и не создает индексируемый SEO-мусор. | P2 |
| Search UX | Поиск должен поддерживать точные, категорийные, feature, use-case и non-product запросы. | Проверять запросы: `SKY 360`, `WING`, `XTAL`, SKU/MPN, `Mesh`, `Racer`, `11D`, `home office`, `доставка`, `гарантия`. | Поиск не отправляет пользователя в пустоту, если нужная страница или товар есть на сайте. | P2 |
| Search results | Если поиск ведет на список результатов, нужны подсказки и быстрые переходы. | Для категорийных запросов вести на серию или показывать серию первым результатом; для `доставка` и `гарантия` вести в FAQ/contact. | Пользователь видит релевантный товар, серию или справочную страницу в первых результатах. | P2 |
| PDP gallery | Изображения должны отвечать на вопросы о размере, посадке, материале и механизмах. | На товарной странице нужны студийные фото, фото в масштабе, фото материала, механизмов, подлокотников, спинки и базы. | Галерея помогает понять, как кресло выглядит в реальном рабочем окружении. | P0 |
| PDP product facts | Ключевые факты должны быть рядом с решением о покупке, а не только глубоко в тексте. | Цена, наличие, гарантия, доставка, SKU/MPN/GTIN, материал, регулировки и механизм выводить в первом товарном блоке или рядом с ним. | Пользователь не ищет базовые факты по всей странице. | P0 |
| PDP trust blocks | Условия покупки должны быть видны до глубокого чтения. | Incentives: доставка, гарантия, возврат, оплата. Значения синхронизировать с `/faq/` и schema.org. | Перед CTA понятно, что будет после обращения или заказа. | P0 |
| Reviews | Отзывы усиливают доверие только если они реальные, видимые и связаны с конкретным товаром. | Использовать только approved отзывы из Netlify Database и не выводить `AggregateRating` без публичных отзывов. | Нет фейковых рейтингов и schema-расхождений. | P3 |
| Forms | Форма должна быть понятной до отправки и после ошибки. | Сохранять labels, required, native validation; подсказки внутри поля давать через `placeholder`, если выбран такой UX. | Пользователь понимает, что ввести в имя, телефон, email и сообщение. | P2 |
| Checkout/cart | Не внедрять checkout-паттерны без реального checkout-flow. | Shopping cart, checkout forms, order summary и order history оставить вне текущего этапа. | В интерфейсе нет ложных ожиданий полноценной покупки на сайте, если заказ реально идет через обращение. | P3 |

## Проверка После Реализации Блока

| Проверка | Что Смотреть |
|---|---|
| `npm run build` | Сборка Hugo должна проходить без ошибок. |
| Двуязычность | Проверить `uk` и `ru` версии, локализованные ссылки и тексты. |
| Responsive | Проверить mobile и desktop; таблицы, карточки и формы не должны ломать ширину. |
| Product finding | Проверить, что пользователь может найти кресло через главную, mobile menu, каталог, серию, фильтр, поиск и внутренние ссылки. |
| Product list UX | Проверить карточки, фильтры, сортировку, applied filters, количество товаров, empty state и группировку вариантов. |
| Product page UX | Проверить первый экран товара, галерею, изображения в масштабе, характеристики, условия покупки и альтернативы. |
| Accessibility | Проверить labels, focus states, tap targets, aria-label для навигации и отсутствие кликов только по мелкому тексту. |
| SEO | `title`, `description`, visible `H1`, breadcrumbs, canonical, hreflang и schema не должны расходиться. |
| Schema.org | Product, FAQ, ContactPage, Organization и BreadcrumbList должны совпадать с видимым контентом. |
| Core Web Vitals | Не добавлять тяжелый JS и изображения без размеров; LCP-изображения должны иметь правильный priority. |
| Внутренняя перелинковка | Блоки должны вести в каталог, серии, товары, FAQ, контакты или профильные статьи. |
