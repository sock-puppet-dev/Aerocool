# Карта Tailwind Plus UI-Секций Для Aerocool

Обновлено: 2026-06-12.

## Зачем Нужен Этот Документ

Этот документ фиксирует рабочую карту UI-блоков для проекта `Aerocool Ukraine` на базе трех групп Tailwind Plus:

- [Marketing UI Blocks](https://tailwindcss.com/plus/ui-blocks#product-marketing)
- [Application UI Blocks](https://tailwindcss.com/plus/ui-blocks#product-application-ui)
- [Ecommerce UI Blocks](https://tailwindcss.com/plus/ui-blocks#product-ecommerce)

Важно: Tailwind Plus используется здесь как **референс структуры и UX-паттернов**, а не как источник для механического копирования. Реализация должна учитывать текущий стек проекта: `Hugo 0.163.0`, `Tailwind CSS 4.3`, локальные overrides в `layouts/`, контент в `content/`, schema.org partials в `layouts/_partials/_schema` и текущий visual layer в `assets/css/main.css`.

## Базовые Правила Внедрения

| Правило | Что Это Значит Для Проекта |
|---|---|
| Сначала локальные overrides | Правки вносятся в `layouts/`, `content/`, `assets/css/main.css` и `data/`, а не напрямую в `themes/PaperMod`, если это можно решить локально. |
| Tailwind 4.3 CSS-first | Проектные дизайн-токены сначала фиксируются в `@theme`, чтобы они были частью Tailwind-системы. `:root` используется как совместимый слой для PaperMod и существующих CSS-хуков. |
| Цветовые utilities только из проекта | В локальных shortcodes и partials использовать `text-aero-*`, `bg-aero-*`, `border-aero-*`, `ring-aero-*`, `stroke-aero-*`, `fill-aero-*`, а не дефолтные `gray/red/white/black` классы Tailwind, если это не осознанный внешний паттерн. |
| Tailwind Plus не копируется слепо | Берем структуру секции, UX-логику, responsive-поведение и типы элементов, но адаптируем визуальный слой под Aerocool. |
| Единая типографика важнее копирования классов | Для заголовков, lead-текста, карточек, meta, бейджей, alert-состояний, footer, форм и кнопок использовать проектные CSS-хуки `.ui-*` из `assets/css/main.css`; прямые Tailwind `text-*`, `tracking-*` и цветовые классы оставлять только там, где они не ломают общий масштаб. |
| Блок должен решать задачу страницы | Каждый UI-блок должен помогать выбрать кресло, перейти в каталог, сравнить модели, понять условия покупки или связаться с Aerocool. |
| SEO и UX должны совпадать | Видимый текст, внутренние ссылки, product facts и JSON-LD не должны расходиться. |
| Не добавлять тяжелый JS без причины | Для базового каталога, FAQ, контактов, таблиц и карточек сначала использовать статический Hugo/Tailwind. |
| Не плодить SEO-мусорные URL | Фильтры каталога должны быть UX-инструментом, а не генератором лишних индексируемых страниц. |
| Двуязычность обязательна | Если блок появляется в `uk`, должна быть синхронная `ru`-версия, если задача явно не ограничена одним языком. |
| Baymard использовать как UX-контроль | Tailwind Plus задает визуальную структуру, а Baymard помогает проверить e-commerce UX: навигацию, списки товаров, фильтры, поиск, PDP, формы и checkout-границы. |
| Проверять сборку | После реализации запускать минимум локальную проверку Hugo. Для задач, где важно не перезаписать локальные отзывы, использовать `hugo server`; `npm run build` запускать только когда это безопасно для текущего review workflow. Для шаблонов и CSS дополнительно проверять главную, листинг и детальную страницу в обоих языках. |

## Стандарт Ширины Сайта

С 2026-06-04 базовый широкий контейнер проекта — **1440 px**. Это новый стандарт для e-commerce-слоя Aerocool: шапки, каталога, страниц серий, товарных сеток, крупных Tailwind Plus-блоков и будущей структуры с фильтрами.

В `assets/css/main.css` стандарт зафиксирован через токены:

- `--site-width: 1440px` — полезная ширина широкого контейнера.
- `--content-section-width: 1200px` — полезная ширина внутренних информационных разделов: `/about/`, `/articles/`, `/news/`, `/faq/`, `/contact/`.
- `--nav-width: var(--site-width)` — ширина навигации.
- `--main-width: var(--site-width)` — ширина основного широкого контейнера.
- `--gap: clamp(1.25rem, 2.5vw, 2.5rem)` — единый боковой отступ для базовых блоков: примерно от **20 px** до **40 px** с каждой стороны.
- `--text-width: 720px` — читаемая ширина длинного текста; ее не расширять автоматически вместе с каталогом.

Базовый боковой отступ должен приходить из общего контейнера сайта, а не из случайных Tailwind-классов `px-*` внутри каждого блока. Если Tailwind Plus-блок уже находится внутри `.main` или `.post-content`, его внутренний wrapper не должен добавлять второй горизонтальный padding. Если блок намеренно выходит на всю ширину секции, например товарные секции главной страницы, внешний фон может расширяться до `100% + 2 * --gap`, но контент внутри должен начинаться по той же вертикали, что логотип, шапка, каталог и обычное тело страницы.

Для каталога и фильтров ориентир такой: sidebar фильтра примерно **260-300 px**, без отдельного горизонтального зазора между фильтром и товарами; фильтр и сетка товаров должны выглядеть как единый каталоговый блок с разделительной линией справа от фильтра, но без левой внешней рамки фильтра. Оставшаяся область — компактная сетка товарных карточек. Страницы, где такая ширина не нужна, например FAQ, контакты и длинные статьи, можно дополнительно ограничивать внутри блока через более узкие текстовые контейнеры, но внешний ритм отступов должен оставаться единым.

## Текущий Статус На 2026-06-12

Повторная UX/UI-проверка `65` подтвердила выводы полного visual audit `64`: проект уже перешел от PaperMod-блоговой структуры к более управляемой каталоговой структуре на базе локальных Hugo overrides и Tailwind 4.3 visual layer. P1-дефект с неоднозначными root product titles закрыт: общий каталог теперь показывает серию в названиях товаров. После проверки `2026-06-12` дополнительно закрыт data/UI mismatch в цветовых точках карточек: `product_group_id` остается главным источником для реальных групп вариантов, а одиночные товары берут `color` из своей product entity через `about_entities`. До следующей оценки важно закрыть тестовый asset в PDP-галерее, слабые series preview images в каталоге, старый search UI, отсутствие applied filter chips и comparison table.

| PDP product facts | Ключевые факты должны быть рядом с решением о покупке, а не только глубоко в тексте. | Цена, наличие, гарантия, доставка, SKU/MPN/GTIN, материал, регулировки и механизм выводить в первом товарном блоке или рядом с ним. | Пользователь не ищет базовые факты по всей странице. | P0 |
| PDP trust blocks | Условия покупки должны быть видны до глубокого чтения. | Incentives: доставка, гарантия, возврат, оплата. Значения синхронизировать с `/faq/` и schema.org. | Перед CTA понятно, что будет после обращения или заказа. | P0 |
| Reviews | Отзывы усиливают доверие только если они реальные, видимые и связаны с конкретным товаром. | Использовать только approved отзывы из Netlify Database и не выводить `AggregateRating` без публичных отзывов. | Нет фейковых рейтингов и schema-расхождений. | P3 |
| Forms | Форма должна быть понятной до отправки и после ошибки. | Сохранять labels, required, native validation; подсказки внутри поля давать через `placeholder`, если выбран такой UX. | Пользователь понимает, что ввести в имя, телефон, email и сообщение. | P2 |
| Checkout/cart | Не внедрять checkout-паттерны без реального checkout-flow. | Shopping cart, checkout forms, order summary и order history оставить вне текущего этапа. | В интерфейсе нет ложных ожиданий полноценной покупки на сайте, если заказ реально идет через обращение. | P3 |

## Критерии Готовности По Ключевым Блокам

| Блок | Считается Готовым, Когда | Проверка В Коде И Контенте |
|---|---|---|
| Category Previews | Есть `SKY`, `WING`, `XTAL` с фото, кратким отличием и локализованными ссылками. | Проверить главную, `/products/`, `/ru/`, пути серий и alt-тексты. |
| Header Navigation | Серии доступны из desktop и mobile menu без глубокого вложения. | Проверить `layouts/_partials/header.html`: desktop flyout содержит `Весь каталог`, `SKY`, `WING`, `XTAL`; mobile menu содержит отдельную группу каталога, а FAQ/контакты отделены как secondary links. Dropdown языка и desktop-каталог должны закрываться при клике вне меню и по `Escape`. |
| Page Headings | Листинги каталога выглядят как каталог, а не как блоговый архив. | Проверить `layouts/products/list.html`: breadcrumbs, `H1`, описание, отсутствие блоговой meta-строки; на страницах серий есть бейджи `SKY`, `WING`, `XTAL`, затем серый `Весь каталог`. |
| Product Lists | Карточка показывает реальные product facts и не выглядит как блоговая карточка. | Данные берутся из front matter товара и entity registry; нет ручного дублирования цены, наличия и цвета в случайных markdown-блоках. Проверить `showSeriesInTitle`, `color-dots.html`, `product_group_id` для реальных групп и fallback цвета через `about_entities` для одиночных товаров. На hover товарная плитка не двигает сетку, а мгновенно показывает скрытые 4 характеристики: серия, материал, регулировки, механизм. |
| Category Filters | Фильтры совпадают с видимыми атрибутами карточек и не создают индексируемые мусорные URL. | Проверить `/products/` и `/ru/products/`: 12 карточек на старте, выбор `SKY` оставляет 2 товара, выбор `SKY + 11D` оставляет `SKY 360`, `Скинути/Сбросить` возвращает все товары, URL не меняется, empty state доступен. На `/products/<series>/` проверить урезанные фильтры без группы серии. На mobile секции фильтра должны быть закрыты по умолчанию. |
| Product Overview | Первый экран товара содержит галерею, название, цену, наличие, CTA, гарантию и идентификаторы товара. | Видимый блок совпадает с `Product` JSON-LD и merchant facts во front matter. |
| Product Gallery | Есть изображения, которые показывают товар, материал, детали и масштаб. | Проверить `image`, `cover`, `products/gallery.html`, размеры изображений, product LCP preload, lazy/eager и alt по языку страницы. |
| Description Lists | Характеристики структурированы и быстро сканируются. | Повторяемые характеристики лучше выводить helper-ом из front matter, а не ручными списками в каждой странице. |
| Tables | Таблица сравнения читается на desktop и mobile. | На mobile использовать stacked layout или скрывать только вторичные колонки, не критические факты. |
| Search | Поиск находит модели, серии, материалы, сценарии и справочные страницы. | Протестировать запросы на украинском, русском и смешанные брендовые запросы. |
| Contact Form | Форма содержит labels, required, native validation, honeypot, success URL и понятные placeholders. | Проверить `layouts/_shortcodes/contact.html`, success page и Netlify form attributes. |
| Alerts | Success-состояние визуально отличается от обычного текста и содержит действия. | Проверить `layouts/_shortcodes/contact-success-alert.html`, `/contact/success/`, `/ru/contact/success/`; browser validation оставить для ошибок полей. |
| About Components | Верх about-страницы выглядит как управляемая брендово-коммерческая структура, а не как длинный markdown-текст. | Проверить `about-intro.html`, `about-series-preview.html`, `about-ergonomics-features.html`, `about-service-features.html`, `about-product-facts.html`, `about-next-steps.html`, `/about/`, `/ru/about/`, локализованные URL и mobile/desktop. |
| FAQ | Вопросы сгруппированы и видимый FAQ совпадает с `FAQPage` JSON-LD. | Проверить `faq_groups`, `faq-list.html` и `_schema/faq.html`. |
| FAQ Related Links | Блок следующих шагов после FAQ выглядит как управляемый UI-компонент, а не как markdown-список. | Проверить `layouts/_shortcodes/recommended-links-faq.html`, `/faq/`, `/ru/faq/`, mobile/desktop, локализованные URL и отсутствие лишних внешних изображений. |
| News Section Highlights | Блок “Что уже обновлено” выглядит как feature-grid, а не как markdown-список. | Проверить `layouts/_shortcodes/section-highlights-news.html`, `/news/`, `/ru/news/`, 6 пунктов, mobile/desktop и отсутствие JS. |
| News Recommended Links | Рекомендованные новости выглядят как управляемый UI-компонент, а не как markdown-список. | Проверить `layouts/_shortcodes/recommended-links-news.html`, `/news/`, `/ru/news/`, mobile/desktop, локализованные URL и отсутствие внешних изображений из Tailwind-примера. |
| Articles Section Highlights | Блок “Что вы найдете” выглядит как feature-grid, а не как markdown-список. | Проверить `layouts/_shortcodes/section-highlights-articles.html`, `/articles/`, `/ru/articles/`, 5 пунктов, mobile/desktop и отсутствие JS. |
| Articles Recommended Links | Рекомендованные стартовые статьи выглядят как управляемый UI-компонент, а не как длинный markdown-список. | Проверить `layouts/_shortcodes/recommended-links-articles.html`, `/articles/`, `/ru/articles/`, mobile/desktop, 7 ключевых ссылок и локализованные URL. |
| Footer | Footer помогает найти серии, товары, статьи, FAQ, контакты и важные условия. | Не превращать footer в повтор главной; держать структуру компактной и локализованной. |

## Приоритет Внедрения

| Очередь | Блоки | Почему Именно Так |
|---|---|---|
| P0 | Поддерживать уже выполненные product overview, product incentives, product cards, FAQ/contact/about/articles/news blocks | Эти блоки уже стали базой текущего UX/UI и не должны регрессировать при следующих правках. |
| P1 | Applied filter chips, comparison table | Это напрямую улучшает product finding и e-commerce UX без checkout-flow; базовые catalog filters, sorting и mobile navigation уже выполнены. |
| P2 | Search UX, placeholders в forms, footer refinement, series FAQ/materials, product alternatives, multi-select filters | Это важно для качества UX, но не блокирует базовый каталог. |
| P3 | Reviews, quickviews, advanced interactive filters | Это требует подтвержденных данных или дополнительного JS/инфраструктуры. |

| Tailwind Plus Блок | Почему Не Сейчас | Когда Вернуться |
|---|---|---|

| Product Quickviews | Добавляет модалки и JS, а базовый каталог еще важнее. | После стабилизации product cards, product pages и фильтров. |

## Проверка После Реализации Блока

| Проверка | Что Смотреть |
|---|---|
| `hugo server` или `npm run build` | Для задач, где нельзя трогать локальные отзывы, проверять через `hugo server` на локальном порту. `npm run build` запускать, когда review workflow безопасен и нужна полная production-like сборка. |
| Двуязычность | Проверить `uk` и `ru` версии, локализованные ссылки и тексты. |
| Responsive | Проверить mobile и desktop; таблицы, карточки и формы не должны ломать ширину. |
| Product finding | Проверить, что пользователь может найти кресло через главную, mobile menu, каталог, серию, фильтр, поиск и внутренние ссылки. |
| Product list UX | Проверить карточки, hover-детали, фильтры, сортировку, applied filters, количество товаров, empty state и группировку вариантов. |
| Product page UX | Проверить первый экран товара, галерею, изображения в масштабе, характеристики, условия покупки и альтернативы. |
| Accessibility | Проверить labels, focus states, tap targets, aria-label для навигации и отсутствие кликов только по мелкому тексту. |
| SEO | `title`, `description`, visible `H1`, breadcrumbs, canonical, hreflang и schema не должны расходиться. |
| Schema.org | Product, FAQ, ContactPage, Organization и BreadcrumbList должны совпадать с видимым контентом. |
| Core Web Vitals | Не добавлять тяжелый JS и изображения без размеров; LCP-изображения должны иметь правильный priority. |
| Внутренняя перелинковка | Блоки должны вести в каталог, серии, товары, FAQ, контакты или профильные статьи. |
